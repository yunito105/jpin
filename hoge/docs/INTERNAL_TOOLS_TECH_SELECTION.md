## 前提 — 既存の EC サイト向けガイドとの違い

[frontend-development-guide.md](frontend-development-guide.md) は**顧客向け EC サイト**（SEO・ISR・公開 URL が重要）を前提としている。
本書は**社内向けツール群**を対象とし、以下の点が異なる。

| 観点 | EC サイト（既存ガイド） | 社内ツール（本書） |
|---|---|---|
| ユーザー | 一般消費者（不特定多数） | 店舗従業員・本部スタッフ（数百〜数千名） |
| SEO | 必須（商品ページ） | **不要**（認証必須のクローズド環境） |
| レンダリング | SSR + RSC 推奨 | **SPA（CSR）が第一選択肢** |
| リアルタイム性 | △（在庫は ISR で許容） | **◎（売り場マップ・作業計画の即時反映）** |
| オフライン対応 | 不要 | △（店舗 Wi-Fi 障害時の考慮） |
| 画面数 | 大（商品一覧・詳細・カート…） | 中（サブシステムごとに独立） |
| UX 要件 | ブランド体験・購買導線 | **業務効率・一覧性・操作速度** |
| FW | Next.js (SSR/RSC) | **Vite + React (SPA)** |
| UI ライブラリ | 自由（shadcn/ui 等） | **MUI（確定）** |
| 業務ロジック | FE にも一部存在 | **BE（Spring Boot）に集約、FE に漏洩させない** |
| アーキテクチャ | Pages Router / App Router | **Feature-Sliced Design (FSD)** |

### 設計原則 — フロントエンド・バックエンド責務分離

```
│  │                          │  │                          │    │

│               │◄──────────────────────────►│                    │
│               │  JSON (OpenAPI 定義)        │                    │
│                                                                │
│  【禁止事項】                                                    │

│  ✕ FE で直接 DB アクセスしない                                    │
│  ✕ FE に業務ルール（割引条件・承認フロー等）をハードコードしない     │

│  ○ UI 状態の制御（ローディング・エラー表示・モーダル開閉）          │
│  ○ 楽観的更新（UX 向上のための一時的な状態変更）                    │
└────────────────────────────────────────────────────────────────┘
```

## 1. リポジトリ構成 — 単一リポジトリ（確定）

### 1.1 リポジトリ構成の比較と本PJ の判断

| パターン | 役割 | メリット | デメリット | 判断基準 |
|---|---|---|---|---|
| **A: Monorepo** | 1リポジトリに複数アプリ + 共通パッケージを格納 | ・共通コンポーネント・型を複数アプリで共有可能<br>・横断リファクタリングが1PRで完結 | ・リポジトリが大きくなると CI が遅くなる<br>・初期セットアップがやや複雑<br>・**サブシステムを独立デプロイできる反面、本PJでは不要な複雑さ** | ・FE が明確に独立した複数のデプロイ単位に分かれる場合<br>・本PJでは不採用 |
| **B: Polyrepo** | サブシステムごとに独立リポジトリ | ・チームの独立性が最大<br>・デプロイが完全独立 | ・共通処理のバージョン管理が煩雑<br>・横断変更に複数 PR が必要 | ・チームが地理的に分散し独立リリースが最優先の場合<br>・本PJでは不採用 |
| **C: 単一リポジトリ（確定）** | 1つの Vite + React SPA に全サブシステムを収録。FSD でページ単位に分割 | ・**構成がシンプル**<br>・**デプロイが1回**（S3 + CloudFront など）<br>・`entities/` を全11システムで自然に共有（product・order・store 等）<br>・設定の統一が容易<br>・チームの認知コストが最小 | ・バンドルサイズが肥大化する可能性（→ ドメインクラスタ単位の React.lazy Code Splitting で対策）<br>・機能間の依存が密になりがち（→ FSD の依存ルールで対策） | ・**本PJ採用**<br>・11システムが同じKeycloakで同じ店舗スタッフが利用<br>・エンティティ共有の恩恵が大きい<br>・デプロイ単位が1つで十分 |

```
│   │   ├── work/                ← 【作業系】
│   │   │   ├── WorkPlannerPage      作業計画システム
│   │   │   └── WorkNavPage          作業ナビシステム
│   │   ├── product/             ← 【商品系】
│   │   │   ├── ProductSearchPage    商品検索システム
│   │   │   ├── DisposalPage         廃棄申請システム
│   │   │   └── StoreTransferPage    店舗間移動システム
│   │   ├── floor-map/           ← 売り場マップシステム
│   │   ├── skill-board/         ← スキルボードシステム
│   │   └── daily-ops/           ← 日次開始・終了システム
│   ├── widgets/                 ← FSD: ページをまたぐ独立 UI 塊
│   ├── features/                ← FSD: ユーザー操作単位のロジック
│   ├── entities/                ← FSD: BE ドメインモデルの FE 表現（複数システムで共有）
│   └── shared/                  ← FSD: 全体共通（API/UI/lib/config）
├── public/
├── vite.config.ts
├── tsconfig.json
├── package.json                 ← pnpm で管理（単一 package.json）
└── .eslintrc / .prettierrc
```

> **Code Splitting 戦略**: `pages/` 配下のドメインクラスタを `React.lazy` + `Suspense` で遅延読み込み。
> 初期ロードは認証・共通レイアウトのみ。注文担当スタッフは `order/` クラスタだけを読み込む。
> 全11システムを表示しても初期バンドルサイズは変わらない。

> **Monorepo への移行ライン**: 販売ドメイン以外（物流・会計・人事等）の**別ドメインが追加**される、または**チームが独立リリースを要求**し始めた時点で Monorepo（`apps/sales`, `apps/logistics`, ...）を検討する。

### 1.3 アプリケーション内部アーキテクチャ — FSD vs 他パターン

> **単一 Vite + React SPA 内部のディレクトリ設計。**

#### パターン比較

| パターン | 概要 | メリット | デメリット | 判断基準 |
|---|---|---|---|---|
| **Feature-Sliced Design (FSD)** | レイヤー（app → pages → widgets → features → entities → shared）で分割。各レイヤーは下位のみ参照可能 | ・**依存方向が厳密**（上位→下位の一方向）<br>・スケールしても構造が崩れにくい<br>・チームメンバーが「何をどこに置くか」迷わない<br>・業務ドメイン（entities）と UI（widgets）が明確に分離<br>・バックエンドの業務ロジック漏洩を構造的に防げる | ・学習コスト（レイヤー・スライスの概念理解）<br>・小規模アプリにはやや重厚<br>・日本語情報がまだ限定的<br>・厳密なルールがチームの柔軟性を制約する場合あり | ・**本PJ推奨**<br>・4サブシステム × 複数チームで統一的なルールが必要<br>・BE との責務分離を構造で強制したい |
| **シンプル Feature-based** | `features/order/`, `features/planner/` のようにドメインで平坦に分割 | ・理解が容易<br>・小規模で素早く開始可能<br>・制約が少なく柔軟 | ・機能間の依存が暗黙的になりがち<br>・スケール時に構造が崩壊しやすい<br>・共通コードの置き場が曖昧 | ・プロトタイプ・小規模ツール<br>・チーム1-2名の場合 |
| **Atomic Design** | atoms → molecules → organisms → templates → pages で UI を分割 | ・UI コンポーネントの粒度が明確<br>・デザインシステムとの親和性が高い | ・**UI の構造しか定義しない**（データ取得・状態管理の指針なし）<br>・molecules と organisms の境界が曖昧<br>・業務ロジック配置のルールがない | ・デザインシステムの構築に特化したい場合<br>・FSD の shared/ui 内部で部分的に採用は可 |
| **Clean Architecture (Ports & Adapters)** | ドメイン層 → ユースケース層 → インフラ層。依存性逆転の原則を適用 | ・ドメインがフレームワークに依存しない<br>・テスタビリティが高い<br>・バックエンド出身者に馴染みやすい | ・**フロントエンドには過剰**（React 自体がフレームワーク依存）<br>・抽象化層が多くコード量が増える<br>・React のコンポーネントモデルと噛み合わないケースあり | ・FE に複雑な業務ロジックがある場合（本PJではBEに集約するため不要）<br>・バックエンドとの対称性を重視する場合 |

```
    │   └── ...                  │   ※ ユーザー操作に対応する「ユースケース」単位
    │
    ├── entities/                ← 【Layer 5: Entities】※ 複数システムで共有されるほど旨味が増す
    │   ├── order/               │   注文（注文系3システムで共有）
    │   ├── product/             │   商品（注文・商品検索・廃棄・移動で共有）
    │   ├── store/               │   店舗（売り場マップ・店舗間移動・日次開始で共有）
    │   ├── employee/            │   従業員（スキルボード・作業計画で共有）
    │   ├── work-task/           │   作業タスク（作業計画・作業ナビで共有）
    │   ├── skill/               │   スキル評価（スキルボード専用）
    │   ├── inventory/           │   在庫（廃棄申請・店舗間移動で共有）
    │   ├── customer/            │   顧客（注文・注文アフター対応で共有）
    │   └── daily-record/        │   日次記録（日次開始・終了専用）
    │                            │   ※ BE の DTO/Entity に対応する FE 表現
    │
    └── shared/                  ← 【Layer 6: Shared】
        ├── api/                 │   API クライアント基盤（openapi-fetch + orval 生成コード）
        ├── ui/                  │   汎用 UI（MUI ラッパー・共通レイアウト・ローダー）
        ├── lib/                 │   ユーティリティ（日付変換・数値フォーマット等）
        └── config/              │   環境変数・定数・i18n 設定
```

#### FSD の依存ルール

```
  app  →  pages  →  widgets  →  features  →  entities  →  shared
  ─────────────────────────────────────────────────────────────→
                     参照可能な方向（上位 → 下位のみ）

  ✕ entities が features を import してはいけない
  ✕ features が widgets を import してはいけない
  ✕ shared が他のどのレイヤーも import してはいけない
```

#### 本PJ における FSD 適用方針

| 方針 | 内容 |
|---|---|
| **entities は BE のドメインモデルの FE 表現** | Spring Boot の Entity/DTO に対応する TypeScript 型・API hooks・UI カードを格納。**業務ロジック（計算・判定）は含めない** |
| **features はユーザー操作単位** | 「注文作成」「フィルタリング」など。API 呼び出し（TanStack Query の mutation）はここに配置 |
| **widgets はページ内の独立した UI 塊** | MUI DataGrid を使ったテーブル、ダッシュボードカード群など |
| **shared/ui は MUI ラッパー** | MUI コンポーネントをプロジェクト固有にラップ（テーマ統一・共通 Props 定義） |
| **業務バリデーションは BE** | FE の features 内バリデーションは形式チェック（必須・文字数・メール形式）のみ |

## 2. フレームワーク / メタフレームワーク選定

### 2.1 社内ツールにおける選択肢

> **EC サイトとの最大の違い**: SEO が不要なため、SSR の必要性が大幅に低下する。
> 本PJでは**Next.js を使用せず、Vite + React (SPA)** で統一する。
> バックエンドは Spring Boot + Java の REST API であり、BFF 層は設けない。

```
               社内ツール フレームワーク選定フロー

                    ┌───────────┐
                    │ SEO必要？  │
                    └─────┬─────┘
                     No ← │ → Yes（→ EC向けガイド参照）
                          │
                    ┌─────▼─────┐
                    │ BFF層   │
                    │ 必要？    │
                    └─────┬─────┘
                     No ← │ → Yes（Next.js 検討）
                          │
                ┌───────▼───────┐
                │ Vite + React   │ ← 本PJ採用
                │ (SPA)          │
                │ + React Router │
                └───────────────┘
```

| パターン | GitHub ⭐（2026年3月時点） | 役割 | メリット | デメリット | 判断基準 |
|---|---|---|---|---|---|
| **Vite + React（SPA）** | ⭐ ~71,000（Vite）/ ~233,000（React） | Vite をビルドツール、React を UI ライブラリとした純 SPA 構成 | ・起動・HMR が最速<br>・社内ツールの SPA に最適（不要な SSR がない）<br>・構成がシンプル<br>・バンドルサイズを最小化できる<br>・**Spring Boot REST API との分離構成に最適** | ・BFF が必要な場合は別途サーバーが要る<br>・ルーティング・データフェッチは自前構成 | ・**本PJ採用（確定）**<br>・SSR 不要の社内ツール<br>・Spring Boot との FE/BE 完全分離 |
| **Next.js（App Router）** | ⭐ ~129,000 | React ベースのフルスタック FW。SSR/SSG/ISR/RSC を統合提供 | ・Server Actions で軽量 BFF が組める<br>・Route Handler で API プロキシが容易 | ・社内ツールに SSR はオーバースペック<br>・App Router の Server/Client 境界の理解が必須<br>・バンドルサイズがやや大きい<br>・**Spring Boot と役割が重複**（BFF不要のため） | ・本PJでは不採用<br>・BFF 層が必要な場合にのみ検討 |
| **TanStack Router + Vite** | ⭐ ~9,000（TanStack Router） | 型安全なファイルベースルーティング + Vite SPA | ・**型安全性が最高**（パス・パラメータ・検索パラメータすべて型付き）<br>・ローダー内蔵でデータフェッチとルーティング統合 | ・エコシステム・実績が限定的<br>・日本語情報が少ない | ・将来的な移行先として注視 |
| **React Router v7** | ⭐ ~31,000 | React Router v7 に統合。Web 標準重視の FW | ・フォーム処理が Web 標準ベースで直感的<br>・ネストルーティングが強力 | ・エコシステムが小さい<br>・SPA モードは限定的 | ・**Vite + React 構成でのルーターとして採用** |

### 2.2 サブシステム別 推奨フレームワーク

| サブシステム | 推奨 | 理由 |
|---|---|---|
| **店舗注文システム** | **Vite + React (SPA)** | 在庫 API・商品マスタは Spring Boot REST API 経由。SPA で十分 |
| **作業計画ツール** | **Vite + React (SPA)** | D&D 中心で SPA 向き。リアルタイムは SSE/WS で対応 |
| **売り場マップ** | **Vite + React (SPA)** | Canvas / SVG 描画中心。SSR のメリットなし。パフォーマンス最優先 |
| **スキルボード** | **Vite + React (SPA)** | シンプルな CRUD + チャート。SPA で完結 |

> **結論**: 全サブシステムを **Vite + React (SPA)** で統一。
> バックエンドは Spring Boot REST API に集約し、BFF 層は設けない。
> 業務ロジックはすべて BE 側、FE は UI 描画・ユーザー操作・状態管理に専念。

### 2.3 なぜ Next.js ではないのか

> **Next.js は GitHub ⭐ ~129,000。圧倒的に人気。ではなぜ使わないのか？**

| 観点 | Next.js | Vite + React (SPA) | 本PJへの影響 |
|---|---|---|---|
| **SSR** | ✅ 組み込み | ✕ なし | ← **本PJは全画面が認証必須の社内ツール。SEO 不要。SSR の恩恵がない** |
| **BFF 機能** | ✅ Server Actions / Route Handler | ✕ なし | ← **BE は Spring Boot。BFF は不要** |
| **バンドルサイズ** | 大（ランタイム含む） | 小（純粋な SPA） | ← 社内ツールは軽いほうが良い |
| **学習コスト** | 高（Server/Client 境界、RSC、App Router） | 低（純粋な React） | ← チームの立ち上げを卓くしたい |
| **Spring Boot との役割** | 重複（Next.js がサーバー側処理を担う） | 明確に分離 | ← **責務分離の原則に反する** |
| **デプロイ** | Node.js サーバーが必要 | S3 + CloudFront だけ | ← 運用がシンプル |

```
【Next.js を採用した場合の問題】

  React SPA (Vite)          Next.js           Spring Boot
    │                         │                   │
    │── REST API ─────────────────────→ │  ← シンプル
    │                         │                   │
                              │─ Server Actions ─→ │  ← Next.js が BFF になる
                              │  Route Handler      │     BEと役割が重複
                              │  API Proxy           │     → 運用コスト増
                              │                   │
  → 本PJ: FE→BE 直結。中間に Next.js を挨む理由がない
```

**Next.js を採用すべきケース**（参考）: EC サイト（SEO 必須）、BFF レイヤーが必要（複数 BE の集約）、Node.js BE を採用する場合

### 2.4 なぜ TypeScript 確定なのか

> **TypeScript は本PJで確定。純粋な JavaScript は使用しない。**

| 観点 | JavaScript | TypeScript | 本PJへの影響 |
|---|---|---|---|
| **型安全** | ✕ ランタイムまでエラーがわからない | ✅ コンパイル時にエラー検知 | ← **11システム × 複数エンティティ共有。型なしは危険** |
| **OpenAPI 連携** | ✕ 型生成の恩恵が半減 | ✅ orval が生成した型をそのまま使える | ← **Spring Boot OpenAPI → TS 型のパイプラインが成立** |
| **リファクタリング** | ✕ 影響範囲が見えない | ✅ 型を変えれば全影響箇所がエラーに | ← **11システムの横断変更を安全に** |
| **チーム開発** | IDE 補完が弱い | ✅ IDE の補完・ナビゲーションが強力 | ← 複数人が同じコードベースで作業 |
| **MUI / TanStack Query** | 型補完が効かない | ✅ Props ・レスポンス型が自動補完 | ← 開発速度が大幅向上 |

```typescript
// TypeScript の恩恵 — entities 共有が安全になる

// entities/order/model.ts — BE の DTO から orval が自動生成
interface Order {
  id: string
  status: 'PENDING' | 'CONFIRMED' | 'CANCELLED' | 'AFTER_CARE'
  items: OrderItem[]
}

// pages/order/  ← 店舗注文システムで使う
// pages/order/  ← 注文キャンセルシステムでも使う
// pages/order/  ← アフター対応システムでも使う
// → 型が共有されているので、BE の DTO 変更時に全システムでコンパイルエラー
```

### 4.1 なぜ Tailwind CSS / shadcn/ui / Ant Design ではないのか

> **Tailwind CSS（⭐ ~86,000）と shadcn/ui（⭐ ~82,000）は人気が高いが、本PJでは採用しない。**

#### Tailwind CSS が合わない理由

```
【MUI と Tailwind のスタイリング競合】

  MUI:     Emotion (CSS-in-JS) でスタイル適用
  Tailwind: ユーティリティクラスでスタイル適用

  併用すると:
  <Button className="bg-blue-500 p-4">
    ↑ Tailwind のクラスが MUI の Emotion スタイルと競合。
    ↑ どちらが優先されるか不定。デバッグが困難に。
    ↑ MUI のテーマシステム（sx prop、styled API）が使えなくなる。
```

| 観点 | MUI (sx prop) | Tailwind CSS | 本PJの判断 |
|---|---|---|---|
| **テーマ統一** | ✅ MUI テーマの `spacing`, `palette` が自動適用 | ✕ MUI テーマと別管理 | MUI のテーマ一元管理が消える |
| **コンポーネント型** | ✅ Props に型がある | ✕ className 文字列に型なし | TypeScript の恩恵が半減 |
| **DataGrid 等のスタイル** | ✅ MUI X と自動統一 | ✕ MUI X のスタイルを Tailwind で上書きできない | 業務ツールの核 = DataGrid。スタイル統一が必須 |

#### shadcn/ui が合わない理由

shadcn/ui は Tailwind CSS + Radix UI ベース。**Tailwind が前提**のため、MUI 採用の本PJでは使用不可。
また、shadcn/ui には MUI X DataGrid 相当の高機能テーブルがない。業務ツールで最も使う DataGrid ・ DatePicker ・ Charts が MUI X でワンストップで揃うのが MUI の強み。

#### Ant Design（⭐ ~93,000）を採用しない理由

| 観点 | MUI | Ant Design | 本PJの判断 |
|---|---|---|---|
| **デザインの柔軟性** | ✅ テーマカスタマイズが自由 | △ Ant のデザインから外れにくい | MUI のほうがカスタマイズしやすい |
| **ドキュメント** | ✅ 英語ファースト | △ 中国語ベース（英語版あり） | チームの一次情報が英語 | 
| **MUI X 相当** | ✅ DataGrid / DatePicker / Charts | ✅ ProTable / ProForm | どちらも充実だが MUI が確定済み |
| **バンドルサイズ** | 大 | さらに大 | Ant Design のほうが重い |

**Ant Design を採用すべきケース**（参考）: 既存プロジェクトで Ant Design が標準化されている場合、中国市場向けの開発で中国語ドキュメントが強みになる場合

### 5.4 なぜ Redux Toolkit ではないのか

> **Redux Toolkit は GitHub ⭐ ~61,000。状態管理のデファクト。ではなぜ使わないのか？**

| **バンドルサイズ** | ~12KB | TanStack Query ~13KB + Zustand ~1KB | ほぼ同等 |

## 6. リアルタイム通信 — 売り場マップ・注文ステータスの即時反映

### 6.1 パターン比較

```
【ポーリング】
  Client ──GET──→ Server    (3秒ごとに繰り返し)
  Client ←─200──  Server
  Client ──GET──→ Server
  Client ←─200──  Server

【SSE (Server-Sent Events)】
  Client ──GET──→ Server    (接続を張りっぱなし)
  Client ←─event─ Server    (サーバーから一方向プッシュ)
  Client ←─event─ Server
  Client ←─event─ Server

【WebSocket】
  Client ←──→ Server       (双方向通信)
         upgrade
  Client ──msg──→ Server
  Client ←─msg──  Server
  Client ──msg──→ Server
```

| パターン | メリット | デメリット | 判断基準 |
|---|---|---|---|
| **ポーリング（Short Polling）** | ・実装最簡単<br>・既存 REST API をそのまま利用<br>・ステートレスでスケールしやすい | ・無駄なリクエストが発生<br>・リアルタイム性が低い（間隔依存）<br>・サーバー負荷が高い | ・更新頻度が分〜10秒間隔で十分な場合<br>・**スキルボード**（日次更新レベル） |
| **SSE（Server-Sent Events）** | ・サーバー→クライアントの一方向プッシュ<br>・HTTP/2 と相性が良い<br>・自動再接続内蔵<br>・REST と同じポートで動作 | ・クライアント→サーバーの送信は別途 REST<br>・接続数に制限あり（ブラウザ6本/ドメイン） | ・**売り場マップ（サーバーからの売上データプッシュ）**<br>・注文ステータスの通知 |
| **WebSocket** | ・双方向通信<br>・低レイテンシ<br>・バイナリデータ対応 | ・接続管理が複雑<br>・ロードバランサーの sticky session 配慮<br>・再接続ロジックが必要 | ・**作業計画ツール（複数管理者の同時編集）**<br>・チャット的な双方向通信が必要な場合 |
| **TanStack Query + ポーリング** | ・TanStack Query の `refetchInterval` で定期更新<br>・既存のキャッシュ管理と統合<br>・実装が最も簡単 | ・リアルタイム性は間隔に依存<br>・積極的な更新には不向き | ・**まずこれで始めて、要件に応じて SSE/WS に移行**<br>・MVP フェーズの推奨 |

### 6.2 サブシステム別推奨

| サブシステム | 推奨方式 | 理由 |
|---|---|---|
| **店舗注文システム** | TanStack Query ポーリング（5秒）→ SSE に移行 | 注文ステータス更新はサーバー→クライアント一方向で十分 |
| **作業計画ツール** | WebSocket（同時編集がある場合）/ SSE（閲覧中心なら） | 複数管理者が同時に計画を編集する場合は双方向が必要 |
| **売り場マップ** | SSE（推奨）or WebSocket | 売上データのリアルタイムプッシュ。サーバー→クライアント一方向で十分 |
| **スキルボード** | TanStack Query ポーリング（30秒〜） | 更新頻度が低い。ポーリングで十分 |

## 11. API 通信

> 本PJ では Spring Boot (Java) の REST API と SPA が直接通信する構成。
> BFF 層は設けず、OpenAPI 定義から型安全に API を呼び出す。
> **業務ロジックはすべて Spring Boot 側。FE は API レスポンスを受けて表示するのみ。**

```
┌──────────────────────────────────────────────────────────────┐
│                     API 通信フロー                           │
│                                                              │
│  React SPA (Vite)         Spring Boot (Java)     Oracle      │
│    │                         │                     │          │
│    │── REST API (JSON) ──→  │                     │          │
│    │   openapi-fetch          │── 業務ロジック  ───→│          │
│    │   + TanStack Query       │   バリデーション     │          │
│    │                         │   トランザクション     │          │
│    │◄─ JSON Response ─────  │                     │          │
│    │                         │                     │          │
│  ↓ 表示のみ                   ↓ 業務の全て                         │
└──────────────────────────────────────────────────────────────┘
```

| パターン | GitHub ⭐（2026年3月時点） | メリット | デメリット | 判断基準 |
|---|---|---|---|---|
| **openapi-fetch** | ⭐ ~2,500（openapi-typescript内） | ・OpenAPI 定義から型安全に API 呼び出し<br>・fetch ベースで軽量<br>・**Spring Boot の OpenAPI 定義と自動連携** | ・OpenAPI 定義が必須<br>・エコシステムが小さい | ・**本PJ推奨**（Spring Boot の springdoc-openapi と連携） |
| **orval** | ⭐ ~3,500 | ・OpenAPI → TanStack Query のコード自動生成<br>・hooks / MSW モック一括生成<br>・型安全 | ・設定ファイルの理解が必要<br>・コード生成に依存 | ・**本PJで有力**（TanStack Query のボイラープレート削減） |
| **fetch API** | — (ブラウザ組み込み) | ・依存ゼロ | ・型安全は自前<br>・エラーハンドリングが冗長 | セopenapi-fetch の内部で使用 |
| **Axios** | ⭐ ~106,000 | ・インターセプターで認証自動付与<br>・広く普及 | ・fetch + TanStack Query と役割が被る<br>・バンドルサイズ増 | ・インターセプターが必要な場合のみ検討 |

### 11.1 なぜ Axios ではないのか

> **Axios は GitHub ⭐ ~106,000。HTTP クライアントで圧倒的一番人気。ではなぜ使わないのか？**

> **Axios は「HTTP クライアント」。openapi-fetch は「型安全な OpenAPI クライアント」。解決する問題のレイヤーが違う。**

```typescript
// ✕ Axios — 型は自己申告。URLはハードコード。
const res = await axios.get<Order>('/api/orders/123')
// ↑ Order 型は自分で書いた型。BE の実際のレスポンスと乖離しても TS は気づかない。
// ↑ URL を /api/v2/orders に変えても型エラーにならない。

// ✅ openapi-fetch — URLもパラメータもレスポンスも全て型安全
const { data, error } = await client.GET('/orders/{orderId}', {
  params: { path: { orderId: '123' } }
  //                ^^^^^^^^ 型エラー: orderId は string（BE の spec から自動生成）
})
// ↑ data の型も自動生成された Order 型。Spring Boot と常に同期。
```

| 観点 | Axios | openapi-fetch + orval | 本PJへの影響 |
|---|---|---|---|
| **URL の型安全** | ✕ 文字列ハードコード | ✅ OpenAPI path から自動生成 | ← 11システム×多数エンドポイント。型なしは危険 |
| **レスポンス型** | 手書き（乖離リスク） | ✅ BE schema から自動生成 | ← **BE 変更時にビルドエラーで検知** |
| **クエリパラメータ** | ✕ 型なし | ✅ 自動生成・typo でコンパイルエラー | ← フィルタ・ページネーションのパラメータが多い |
| **学習コスト** | ほぼゼロ | やや高い（orval の理解） | 型安全の恩恵 > 学習コスト |
| **バンドル** | ~13KB | ~7KB（fetch ベース） | openapi-fetch のほうが軽い |

```
【本PJ の API 通信パイプライン】

  Spring Boot (springdoc-openapi)
    │
    ↓ 自動生成
  openapi.json（API 仕様書 — 唯一の真実）
    │
    ↓ orval が変換（CI/CD に組み込み）
  TypeScript 型 + openapi-fetch クライアント + TanStack Query hooks
    │
    ↓
  FE コード — URLもパラメータもレスポンス型もすべて型安全

  → BE の DTO 変更 → openapi.json 更新 → TS ビルドエラー → FE 修正
  → Axios ではこのパイプラインが成立しない
```

**Axios を採用すべきケース**（参考）: OpenAPI spec がない（BE がレガシー / 仕様書未整備）、既存プロジェクトで Axios が深く使われている、インターセプターを多用した複雑なリクエスト制御が必要

## 12. 認証方式

> 社内ツールは全画面が認証必須。認証はプロジェクト共通基盤で一元管理。
> SPA 構成のため、トークンはブラウザ側で管理（PKCE フロー）。

```
┌────────────────────────────────────────────────────────────┐
│              SPA 認証フロー (PKCE)                           │
│                                                            │
│  React SPA (Vite)         Keycloak         Spring Boot     │
│    │                         │                │             │
│    │── OIDC認証 (PKCE) ──→  │                │             │
│    │← ID Token + Access ───  │                │             │
│    │   Token (memory保持)    │                │             │
│    │                         │                │             │
│    │── REST API 要求 ──────────────────────→  │             │
│    │   (Authorization:        │    Bearer 検証  │             │
│    │    Bearer <token>)       │←────────────  │             │
│    │← JSON Response ────────────────────────  │             │
└────────────────────────────────────────────────────────────┘
```

| パターン | メリット | デメリット | 判断基準 |
|---|---|---|---|
| **Keycloak + OIDC (PKCE)** | ・**SPA に最適な PKCE フロー**<br>・SSO 対応（社内の他システムと統合）<br>・RBAC（ロールベースアクセス制御）内蔵<br>・トークンリフレッシュ自動化 | ・Keycloak の運用コスト<br>・トークンがブラウザ memory に存在（XSS 対策必須） | ・**本PJ推奨**（社内SSO + 複数サブシステムの認証統合） |
| **keycloak-js** | ・公式 JS アダプター<br>・トークン管理・リフレッシュが内蔵 | ・バンドルサイズがやや大きい<br>・React 専用ではない | ・**SPA + Keycloak の標準構成** |
| **oidc-client-ts** | ・軽量な OIDC クライアント<br>・フレームワーク非依存<br>・PKCE / Silent Renew 対応 | ・設定がやや複雑 | ・keycloak-js の代替として検討 |
| **Auth0** | ・マネージドサービスで運用不要<br>・豊富なSPA用SDK | ・外部サービス依存<br>・大規模利用はコスト増 | ・運用チームが小さく Keycloak 運用が困難な場合 |

### 12.1 なぜ Auth0 ではないのか

> **Auth0 はマネージド認証サービスで運用負荷が最小。ではなぜ Keycloak なのか？**

| 観点 | Auth0 | Keycloak | 本PJへの影響 |
|---|---|---|---|
| **運用** | ✅ マネージド（SaaS） | ✕ 自前運用（オンプレ / コンテナ） | 本PJは社内インフラあり。運用可能 |
| **コスト** | 有料（ユーザー数課金） | ✅ OSS 無料 | ← **数千名の店舗スタッフ。Auth0 だとコストが大きい** |
| **社内 SSO** | ✅ 対応 | ✅ 対応 | 同等 |
| **カスタマイズ** | △ 制限あり | ✅ 完全に自由 | ← ログイン画面・ロール定義を完全制御 |
| **データ保管** | 外部（Auth0 サーバー） | ✅ 社内（自社インフラ） | ← **社内ツールの認証情報を外部に出したくない** |

**Auth0 を採用すべきケース**（参考）: スタートアップでインフラ運用チームがいない、ユーザー数が少なくコストが許容範囲の場合

## 19. 判断に迷ったときのフローチャート

```
Q1: フレームワークは？
  └─ Vite + React (SPA) 確定。Next.js は使用しない。
     └─ バックエンド: Spring Boot REST API
        └─ 業務ロジックは BE に集約、FE は表示専念

Q2: UIコンポーネントライブラリは？
  └─ MUI (Material UI) 確定。
     ├─ DataGrid → MUI X DataGrid
     ├─ Form       → MUI TextField + React Hook Form + Zod
     ├─ Chart      → MUI X Charts (基本) / Recharts・ECharts (高度)
     └─ Toast      → MUI Snackbar (+ notistack)

Q3: アプリ内アーキテクチャは？
  └─ FSD (Feature-Sliced Design)
     └─ entities は BE ドメインの FE 表現。業務ロジックを含めない。

Q4: リアルタイム通信は？
  ├─ 更新が分〜10秒間隔で OK → TanStack Query ポーリング
  ├─ サーバー→クライアント一方向 → SSE
  └─ 双方向通信が必要         → WebSocket

Q5: 売り場マップの描画は？
  ├─ 要素数 100 以下      → SVG (React 直接)
  ├─ 要素数 100〜1000     → react-konva (Canvas)
  └─ 要素数 1000 以上     → Pixi.js (WebGL)
```
