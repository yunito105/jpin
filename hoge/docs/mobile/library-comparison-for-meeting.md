# ライブラリ比較・判断表 — 先輩チーム相談用

**作成日**: 2026年3月25日  
**目的**: 各カテゴリの候補ライブラリを比較し、選定の判断基準を明確にした上で会議に臨む

---

## 読み方

各カテゴリで以下を整理している：

1. **なぜ決める必要があるのか**（目的）
2. **候補の比較表**（メリット・デメリット）
3. **判断基準**（何を基準に選ぶか）
4. **自分たちの仮選定と根拠**（会議で伝える内容）

---

## 1. UI フレームワーク

### 目的
画面を何で作るかの根本的な選択。ここを間違えると全コードに影響し、後から変更できない。

### 比較表

| 項目 | **Jetpack Compose** | **XML（View System）** | **Compose + XML 混在** |
|---|---|---|---|
| **公式推奨** | ⭐ Google が推奨。NiA は 100% Compose | メンテナンスモード（新機能追加なし） | 移行期の一時的な状態 |
| **コード量** | 少ない（XML + Binding 不要） | 多い（XML + findViewById or ViewBinding） | 最も多い（両方の知識が必要） |
| **学習コスト** | 高い（宣言的UI の考え方に慣れる必要あり） | 低い（情報が豊富、経験者が多い） | 最も高い（両方を理解する必要あり） |
| **新規プロジェクト向き** | ◎（ゼロから作るなら最適） | △（2025年以降に新規で選ぶ理由が薄い） | △（既存アプリの段階移行向け） |
| **ライブラリ対応** | 主要ライブラリは対応済み | 全ライブラリが対応 | 両方使える |
| **採用企業** | Google, Twitter, Airbnb, LINE など | 既存の大半のアプリ | 移行中のアプリ |
| **テスト** | `ComposeTestRule` で宣言的にテスト可能 | Espresso（記述量が多い） | 混在して複雑 |
| **プレビュー** | `@Preview` で IDE 上で即時確認 | XML プレビューあり | 部分的に可能 |

### 判断基準

| 基準 | 重み | 説明 |
|---|---|---|
| **Google 公式推奨か** | 高 | 長期サポート・情報量・エコシステムの充実度に直結 |
| **新規プロジェクトか** | 高 | 新規なら XML を選ぶ理由がない。既存コードの資産がない |
| **チームの学習余力** | 中 | Compose の学習コストは 2〜4 週間。それを許容できるか |
| **5 年後も使えるか** | 高 | XML は新機能が追加されない。Compose が今後の標準 |

### 仮選定: **Jetpack Compose**

> **先輩に聞くこと**: 「Compose を本番で使っていますか？ 導入時に困ったこと、特に Recomposition やパフォーマンス面で問題はありましたか？」

## 2. DI（依存性注入）

### 目的
クラス間の依存関係を管理する仕組み。ViewModel が Repository を使う、Repository が API クライアントを使う——こうした「誰が誰を使うか」を自動的に解決してくれる。テストの差し替え（本物 → モック）にも必須。

### 比較表

| 項目 | **Hilt** | **Koin** | **手動 DI** |
|---|---|---|---|
| **公式推奨** | ⭐ Google 公式。Jetpack の一部 | コミュニティ製（非公式） | 公式ガイドに記載あり（小規模向け） |
| **エラー検出** | **コンパイル時**（ビルドで発見） | **実行時**（動かして初めて発見） | コンパイル時（手書きなので） |
| **記述量** | やや多い（Module, @Inject 等） | 少ない（DSL で簡潔に書ける） | 最も多い（全部手書き） |
| **ビルド速度** | やや遅い（コード生成が走る） | 影響なし | 影響なし |
| **学習コスト** | 中〜高（Dagger の概念理解が必要） | 低（DSL が直感的） | 低（特別な知識不要） |
| **テスト支援** | `@TestInstallIn` で差し替え容易 | `loadKoinModules` で差し替え | 手動で渡す |
| **大規模対応** | ◎（スコープ管理が型安全） | △（実行時エラーが増えるリスク） | ✕（手書き管理が破綻する） |
| **ViewModel 統合** | `@HiltViewModel` で自動注入 | `viewModel()` 関数で取得 | 手動でファクトリ作成 |
| **NiA での採用** | ✅ Hilt 2.59 + KSP | — | — |

### 判断基準

| 基準 | 重み | 説明 |
|---|---|---|
| **コンパイル時エラー検出** | 高 | 本番で「DI 設定漏れ」でクラッシュするのは許容不可 |
| **大規模プロジェクトへの耐性** | 高 | 画面数が増えた時に破綻しないか |
| **Google 公式サポート** | 中 | ドキュメント・サンプルの充実度 |
| **チームの既存知識** | 中 | Koin 経験者が多いなら Koin の方が立ち上がりが速い |

### 仮選定: **Hilt**

> **先輩に聞くこと**: 「Hilt を使っていますか？ Koin ですか？ Hilt のビルド速度への影響やデバッグの難しさは気になりますか？」

## 3. API 通信

### 目的
バックエンド（Spring Boot）の REST API と通信するためのライブラリ。商品取得・注文送信・認証など、ほぼすべての機能が API 通信を経由する。

### 比較表

| 項目 | **Retrofit2 + OkHttp3** | **Ktor Client** |
|---|---|---|
| **開発元** | Square 社 | JetBrains 社 |
| **公式推奨** | NiA で採用。業界のデファクトスタンダード | Kotlin 公式だが Android 特化ではない |
| **記述スタイル** | インターフェース + アノテーション（`@GET`, `@POST`） | DSL ベース（`client.get { url(...) }`） |
| **Coroutines 対応** | `suspend fun` 対応 | ネイティブ対応 |
| **Interceptor** | OkHttp の Interceptor が強力（ログ・認証・リトライ） | Plugin 機構で同等のことは可能 |
| **情報量** | 非常に多い（10 年以上の歴史） | 少なめ（Android 向け情報が限定的） |
| **JSON 変換** | Converter を差し替え可能（Gson / Moshi / kotlinx.serialization） | kotlinx.serialization がデフォルト |
| **KMP 対応** | ✕（Android / JVM のみ） | ◎（iOS / Web にも対応） |
| **学習コスト** | 中（アノテーションのパターンを覚える） | 中（DSL のパターンを覚える） |
| **エコシステム** | MockWebServer 等テストツールが豊富 | テストツールは Ktor 独自 |

### 判断基準

| 基準 | 重み | 説明 |
|---|---|---|
| **実績・安定性** | 高 | 本番で予期しない不具合が少ないこと |
| **情報の入手しやすさ** | 高 | 困った時に StackOverflow や記事で解決できるか |
| **チームの既存知識** | 中 | Ktor に慣れているチームなら Ktor も悪くない |
| **KMP の予定** | 低 | 将来 iOS と共有する計画があるなら Ktor が有利 |

### 仮選定: **Retrofit2 + OkHttp3**

> **先輩に聞くこと**: 「API 通信は Retrofit ですか？ Ktor を検討したことはありますか？ OkHttp の Interceptor でどんなことをしていますか？（認証トークン自動付与、ログ出力など）」

## 4. ローカル保存

### 目的
2 つの異なる用途がある。混同しないこと。

| 用途 | 説明 | データの例 |
|---|---|---|
| **構造化データの永続化** | テーブル形式で大量データを保存・検索 | 商品一覧キャッシュ、カート内容、注文履歴 |
| **設定値の保存** | キー・バリュー形式で少量のデータを保存 | 言語設定、認証トークン、テーマ |

### 比較表（構造化データ）

| 項目 | **Room** | **SQLDelight** | **Realm** |
|---|---|---|---|
| **開発元** | Google（Jetpack） | Cash App（Square 系列） | MongoDB |
| **公式推奨** | ⭐ Google 公式。NiA で採用 | コミュニティで人気 | 非公式 |
| **記述方法** | Kotlin アノテーション（`@Entity`, `@Dao`） | SQL ファイルから Kotlin コード自動生成 | Kotlin オブジェクト定義 |
| **SQL の知識** | 必要（`@Query` 内に SQL を書く） | 必須（SQL ファイルが中心） | 不要（独自クエリ言語） |
| **Flow 対応** | ◎（`Flow<List<T>>` で DB 変更を自動通知） | ◎ | △（独自のリアクティブ機構） |
| **マイグレーション** | 手動定義（やや煩雑だが安全） | 手動定義 | 自動（制限あり） |
| **KMP 対応** | △（Android のみ） | ◎（iOS / Web にも対応） | ◎ |
| **情報量** | 非常に多い | 中程度 | 中程度 |

### 比較表（設定値）

| 項目 | **DataStore（Preferences）** | **DataStore（Proto）** | **SharedPreferences** |
|---|---|---|---|
| **公式推奨** | ⭐ Google 公式 | ⭐ Google 公式（NiA で採用） | 非推奨（後継が DataStore） |
| **型安全** | △（キーの型は手動管理） | ◎（Protocol Buffers でスキーマ定義） | ✕（String 等で手動管理） |
| **非同期対応** | ◎（Coroutines + Flow） | ◎（Coroutines + Flow） | ✕（メインスレッドをブロック） |
| **学習コスト** | 低 | 中（.proto ファイルの理解が必要） | 低 |
| **用途** | 単純な設定（boolean, string） | 複雑な設定（ユーザー情報、多言語設定） | レガシーアプリの互換用途 |

### 判断基準

| 基準 | 重み | 説明 |
|---|---|---|
| **オフライン対応の必要性** | 高 | オフラインキャッシュが必須なら Room 一択 |
| **Flow との統合** | 高 | DB 変更 → UI 自動更新のパイプラインが組めるか |
| **Google 公式か** | 中 | 長期メンテナンス・Jetpack 統合の安心感 |

### 仮選定: **Room + DataStore（Preferences）**

> **先輩に聞くこと**: 「ローカル DB は Room ですか？ DataStore は Preferences と Proto どちらを使っていますか？ オフラインキャッシュはどの程度やっていますか？」

## 5. 画像読込

### 目的
商品画像をネットワークから取得し、キャッシュして高速に表示する。商品一覧画面で大量の画像を効率的にスクロールさせるために必須。

### 比較表

| 項目 | **Coil** | **Glide** |
|---|---|---|
| **開発元** | Instacart 社 | Google / Bumptech |
| **言語** | Kotlin ネイティブ | Java（Kotlin からも使える） |
| **Compose 対応** | ◎（`AsyncImage` で 1 行） | △（`GlideImage` で対応可能だが公式感は薄い） |
| **キャッシュ** | メモリ + ディスク 2 層キャッシュ | メモリ + ディスク 2 層キャッシュ |
| **APK サイズ影響** | 小（~1,500 メソッド） | 大（~4,500 メソッド） |
| **GIF 対応** | 追加ライブラリが必要 | 標準搭載 |
| **歴史・実績** | 2020 年〜（比較的新しい） | 2014 年〜（10 年以上の実績） |
| **情報量** | 中（増加中） | 非常に多い |
| **NiA での採用** | ✅ Coil 2.7.0 | — |
| **Coroutines 対応** | ネイティブ | Kotlin 向けアダプタ経由 |

### 判断基準

| 基準 | 重み | 説明 |
|---|---|---|
| **Compose との相性** | 高 | Compose を使うなら Coil が自然な選択 |
| **軽量さ** | 中 | APK サイズを抑えたいか |
| **GIF / 動画の必要性** | 低 | 商品画像は静止画が中心ならどちらでも可 |

### 仮選定: **Coil**

> **先輩に聞くこと**: 「画像読込は何を使っていますか？ Coil と Glide で迷ったりしましたか？ 大量画像のスクロール性能で問題はありましたか？」

## 7. JSON パーサー

### 目的
バックエンド API の JSON レスポンスを Kotlin の data class に変換する（デシリアライズ）。逆にリクエストボディを JSON に変換する（シリアライズ）。

### 比較表

| 項目 | **kotlinx.serialization** | **Moshi** | **Gson** |
|---|---|---|---|
| **開発元** | JetBrains（Kotlin 公式） | Square 社 | Google |
| **言語** | Kotlin ネイティブ | Kotlin / Java | Java |
| **コード生成** | コンパイル時（KSP） | コンパイル時（KSP）or リフレクション | リフレクション |
| **Kotlin 対応** | ◎（data class, sealed class, null 安全すべて対応） | ○（Kotlin アダプタで対応） | △（Kotlin の null 安全を無視する場合がある） |
| **速度** | 速い（コンパイル時生成） | 速い（コード生成版） | やや遅い（リフレクション） |
| **APK サイズ** | 小 | 中 | 中 |
| **KMP 対応** | ◎（iOS / Web でも使える） | ✕ | ✕ |
| **学習コスト** | 低（`@Serializable` を付けるだけ） | 中 | 低 |
| **NiA での採用** | ✅ kotlinx.serialization 1.8.0 | — | — |
| **情報量** | 中（増加中） | 多い | 非常に多い |

### 判断基準

| 基準 | 重み | 説明 |
|---|---|---|
| **Kotlin との相性** | 高 | null 安全・sealed class を正しく扱えるか |
| **コンパイル時の安全性** | 高 | リフレクションではなくコード生成で型チェックされるか |
| **Retrofit との統合** | 中 | Converter が用意されているか（3 つとも対応済み） |

### 仮選定: **kotlinx.serialization**

> **先輩に聞くこと**: 「JSON パーサーは何を使っていますか？ Gson からの移行を検討していたりしますか？ kotlinx.serialization で困ったケースはありますか？」

## 8. テスト

### 目的
コードが正しく動くことを自動的に検証する仕組み。テスト戦略は 3 つのレイヤーに分かれる。

| レイヤー | 何をテストするか | 実行場所 |
|---|---|---|
| **Unit Test** | ViewModel, Repository のロジック | PC 上（JVM）で高速実行 |
| **Flow テスト** | StateFlow の状態遷移（Loading → Success → Error） | PC 上（JVM） |
| **UI テスト** | 画面の表示・操作 | エミュレータ or 実機 |

### 比較表（テストフレームワーク）

| 項目 | **JUnit4** | **JUnit5** |
|---|---|---|
| **公式対応** | ⭐ Android 公式。NiA で採用 | android-junit5 プラグインが必要 |
| **Instrumented Test** | ◎（Android テストの基盤） | △（完全対応していない） |
| **Robolectric** | ◎ | △（互換性問題あり） |
| **記法** | `@Before` / `@After` / `@Test` | `@BeforeEach` / `@AfterEach` / `@Test` |
| **ネスト** | `@RunWith(Enclosed::class)` で可能 | `@Nested` で自然に書ける |
| **パラメータテスト** | `@Parameterized` （やや煩雑） | `@ParameterizedTest`（簡潔） |
| **情報量** | 非常に多い（全 Android サンプルが JUnit4） | Android 向けは限定的 |

### 比較表（モッキング）

| 項目 | **MockK** | **Fake 実装（手書き）** | **Mockito** |
|---|---|---|---|
| **NiA の方針** | — | ⭐ NiA は Fake を使用 | — |
| **Kotlin 対応** | ◎（suspend, data class, object 対応） | ◎（純 Kotlin） | △（Kotlin 特有の機能に弱い） |
| **型安全** | 実行時（リフレクション） | コンパイル時（手書きなので） | 実行時 |
| **Interface 変更時** | テストは壊れない（モックが吸収） | テストが壊れる（= 気づける） | テストは壊れない |
| **記述量** | 少ない（DSL で簡潔） | 多い（インターフェースを全実装） | 中 |
| **学習コスト** | 中（MockK DSL の習得） | 低（特別な知識不要） | 中 |
| **テスト品質** | △（実装詳細に依存しがち） | ◎（振る舞いベースのテスト） | △ |

### 判断基準

| 基準 | 重み | 説明 |
|---|---|---|
| **Android 公式との整合性** | 高 | Instrumented Test, Robolectric と問題なく使えるか |
| **Interface 変更の追従** | 高 | Repository が変わった時にテストが壊れて気づけるか |
| **チームの既存知識** | 中 | JUnit5 経験者がいるか、MockK に慣れているか |

### 仮選定: **JUnit4 + Fake 実装 + Turbine**

NiA の方針に合わせ、Repository テストは Fake 実装を基本とする。MockK は補助的に使用。

> **先輩に聞くこと**: 「テストフレームワークは JUnit4 ですか？ JUnit5 ですか？ Repository のテストは MockK でモックしていますか？ それとも Fake 実装を手書きしていますか？ その選択の理由は何ですか？」

### 仮選定: **Spotless + KtLint**（+ 必要に応じて detekt）

> **先輩に聞くこと**: 「コードフォーマッタは何か使っていますか？ CI でフォーマットチェックしていますか？」
