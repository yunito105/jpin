# 先輩チームへの技術相談 — 会議アジェンダ

**日付**: 2026年3月  
**目的**: モバイルサンプルプロジェクト作成にあたり、実開発チームの知見を得る  
**所要時間**: 60分

---

## 前提情報

| 項目 | 内容 |
|---|---|
| **自分の立場** | モバイル実務未経験のエンジニア。サンプルプロジェクトを作る役割 |
| **相手** | モバイル開発を実際に進めている別チームの先輩 |
| **決定済み** | Android のみ / Kotlin / ネイティブ開発（React Native・WebView は使わない） |
| **決めたいこと** | IDE → バージョン管理 → SDK → ライブラリ → アーキテクチャ |

---

## 進行の原則

**大きな枠組み → 具体的な技術選定** の順番で進める。  
いきなり「このライブラリどうですか？」と聞いても文脈がないので伝わらない。

---

## アジェンダ

### 1. 導入・背景共有（5分）

自分たちのプロジェクトの前提を簡潔に伝える。相手に「何のために聞いているのか」をわかってもらう。

**伝えること**:

- 小売業向けの Android アプリを新規で作る
- Kotlin ネイティブで、React Native や WebView は使わない
- まずサンプルプロジェクトを作って、技術選定の土台にしたい
- 自分はモバイル実務経験がないので、実際に開発されている立場からアドバイスをいただきたい

---

### 2. 先輩チームの現状を聞く（10分）

こちらの話をする前に、**相手の環境を先に聞く**。これが後の質問すべての土台になる。

**質問リスト**:

- 今のプロジェクトではどんなアプリを作っていますか？（規模感・画面数）
- チーム人数と、モバイル専任は何人くらいですか？
- Android Studio で開発していますか？ バージョンはどのくらいで追従していますか？
- 対象の Android バージョン（minSdk）はいくつにしていますか？ それはどう決めましたか？

**狙い**: IDE と SDK バージョンの話を「相手の実体験」から自然に引き出す

---

### 3. 開発環境まわり（10分）

先輩の回答を踏まえて、自分たちの環境をどうするか相談する。

| 項目 | 聞きたいこと |
|---|---|
| **IDE** | Android Studio の推奨バージョン。Stable を使うか、Beta/Canary は使うか |
| **Kotlin バージョン** | 2.x 系を使おうと思っているが、実際に 2.x で困ったことはあるか |
| **Gradle / AGP** | バージョンの組み合わせで罠はあるか |
| **minSdk** | 最低 API レベルの決め方（ユーザーシェアを見るか、社内端末に合わせるか） |
| **targetSdk** | Google Play の要件に合わせているか |

**聞き方のコツ**:  
> 「公式ドキュメントだと minSdk 26（Android 8）以上が推奨と見たのですが、実際にはどうされていますか？」  

のように、**調べた上で聞く**と会話が噛み合いやすい。

---

### 4. アーキテクチャの大枠（15分）★ 最重要

**ライブラリの前に、まず設計方針を聞く。** ここが一番重要なパート。

**質問リスト**:

- 画面の設計パターンは何を使っていますか？（MVVM？ MVI？ それとも別のもの？）
- Google の公式ガイドにある ViewModel + Repository のパターンをそのまま使っていますか？
- UseCase 層（Domain 層）は作っていますか？ 作っているならどういう時に使いますか？
- フォルダ構成はどうしていますか？ feature ごとに分けていますか？
- マルチモジュールにしていますか？ それとも単一モジュールですか？

**狙い**: MVVM + Repository を自分たちも検討していることを伝えつつ、実運用での感想を聞く

**深堀り質問**（余裕があれば）:

- オフライン対応はしていますか？ している場合、同期の仕組みはどうしていますか？
- 状態管理は StateFlow ですか？ sealed class/interface で UiState を定義していますか？
- テストはどのくらい書いていますか？ Mock と Fake どちらを使っていますか？

---

### 5. ライブラリ選定（15分）

アーキテクチャの話が済んだ後で、ライブラリの話に入る。  
**先輩が使っている構成を先に聞いてから、自分たちの選定案をぶつける。**

**聞き方**:
> 「具体的なライブラリの話を聞かせてください。以下のカテゴリで何を使っていますか？」

| カテゴリ | 聞きたいこと |
|---|---|
| **UI** | Jetpack Compose？ まだ XML も混在している？ |
| **DI** | Hilt？ Koin？ 手動 DI？ |
| **API 通信** | Retrofit？ Ktor？ |
| **ローカル保存** | Room？ DataStore？ |
| **画像読込** | Coil？ Glide？ |
| **ナビゲーション** | Navigation Compose？ 独自？ |
| **JSON** | kotlinx.serialization？ Moshi？ Gson？ |
| **テスト** | JUnit4？ JUnit5？ MockK？ Fake 実装？ |
| **コードフォーマッタ** | Spotless + KtLint？ 他？ |

**自分たちの案を伝える場合**:
> 「Google の公式サンプル（Now in Android）を参考に、Jetpack Compose / Hilt / Retrofit / Room あたりを使おうと考えているのですが、実際に使ってみて困った点はありますか？」

**狙い**: 「このライブラリどうですか？」ではなく、**「実際に使って困ったことは？」** と聞くことで実務の知見を引き出す

---

### 6. まとめ・次のアクション確認（5分）

- 聞いた内容を簡単に復唱して認識合わせ
- 「サンプルプロジェクトを作ったら見ていただけますか？」と次の接点を作る
- 参考にしているリポジトリやドキュメントがあれば教えてもらう

---

## 会議で意識すること

| ✅ やるべきこと | ❌ やらないこと |
|---|---|
| 相手の構成を**先に聞く** | いきなり自分たちの選定案を説明する |
| 「困ったこと・ハマったこと」を聞く | 「このライブラリいいですか？」と○×を求める |
| 知らない用語が出たらその場で聞く | 知ったふりをする |
| メモを取り、後で方針書に反映する | 全部その場で決めようとする |
| 具体例で質問する（「minSdk 26 にしようと思うのですが」） | 抽象的に聞く（「バージョンどうしてますか？」） |
