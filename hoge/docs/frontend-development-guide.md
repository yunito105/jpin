## 1. ゴール設定 — フェーズ別の完了定義

「何をもって仕事が完了したと言えるか？」をフェーズごとに明確化する。確認先は PM → テックリード上長の順。

| パターン | 役割（何のためにやるか） | メリット | デメリット | 判断基準 |
|---|---|---|---|---|
| **要件定義フェーズのゴール: 「FE 開発を開始できる状態」** | 画面仕様書・デザイン・対象ブラウザ・見積もり・スケジュールが揃っている状態をゴールにする | ・開発開始後の手戻りが最小限<br>・見積もり精度が高くなる<br>・デザイナー/BE チームとの合意が取れている | ・要件が固まらないと開発に着手できない<br>・完璧を求めると時間がかかりすぎる | ・**本PJ推奨**<br>・25億規模では要件定義の不備が数千万単位のロスに直結<br>・「80% 決まったら着手、残 20% は並行で詰める」が現実的 |
| **要件定義フェーズのゴール: 「見積もりができる状態」** | 画面仕様書と主要な技術課題が判明している程度。細部は開発中に詰める | ・着手が早い<br>・アジャイル的に素早く動ける | ・後半で仕様変更が頻発するリスク<br>・見積もりが大きくブレる可能性 | ・小規模 PJ やプロトタイプフェーズ向け<br>・25億規模では仕様の「あいまいさ」がリスクに直結 |
| **開発フェーズのゴール: 「API 結合テスト完了 + バグ全件クローズ」** | デプロイ済み + テスト済み + バグ対応済みの状態をゴールにする | ・リリース品質が担保される<br>・ゴールが明確で進捗管理しやすい | ・バグのクローズ基準を事前に決めないと揉める<br>・「全件」の定義が曖昧だと終わらない | ・**本PJ推奨**<br>・Critical / Major は全件クローズ、Minor は次スプリントでもOK等の基準を事前合意 |
| **開発フェーズのゴール: 「デプロイ完了」** | 動くものがデプロイされていれば OK。バグは後続で対応 | ・スピード重視で回せる<br>・MVP やβ版に向いている | ・品質が担保されない<br>・後半でバグが溢れる可能性 | ・β版リリースやユーザーテスト段階のみ |

## 2. 開発工数見積もり — バッファの取り方

| パターン | 役割 | メリット | デメリット | 判断基準 |
|---|---|---|---|---|
| **1.5倍バッファ** | 見積もり工数に1.5倍の係数をかける | ・多くの現場で実績ある安全係数<br>・不測の事態（仕様変更・バグ）に対応できる | ・顧客から「高い」と言われる可能性<br>・バッファがあると安心して遅れる（パーキンソンの法則） | ・**本PJ推奨**（新規チーム + 大規模PJ）<br>・チーム習熟度が不明な場合<br>・認証・決済・多言語対応など不確実性が高い要素がある |
| **1.2倍バッファ** | 見積もり工数に1.2倍の係数をかける | ・コストが抑えられる<br>・適度な緊張感を維持 | ・想定外の事態に弱い<br>・炎上リスクがやや高い | ・チームが同技術スタックで実績があり、習熟度が高い場合<br>・類似プロジェクトの経験がある場合 |
| **2.0倍バッファ** | 見積もり工数に2.0倍の係数をかける | ・未経験技術でも安全<br>・品質改善の余裕がある | ・コストが大きく膨らむ<br>・スケジュールが間延びする | ・チーム全員が該当技術スタック未経験<br>・外部連携が多く、相手側の遅延リスクが高い場合 |

## 3. フレームワーク選定

| パターン | 役割（どんなフレームワークか） | メリット | デメリット | 判断基準 |
|---|---|---|---|---|
| **Next.js** | React ベースのフルスタック FW。SSR/SSG/ISR/App Router を統合提供 | ・SSR + CSR のハイブリッドが容易<br>・Vercel による手厚いサポート<br>・日本語情報・採用実績が豊富<br>・API Routes で軽量な BFF 構成も可能 | ・Vercel 依存が気になるケースあり<br>・App Router の学習コストがやや高い<br>・バージョンアップの頻度が高い | ・**本PJ確定**（SEO + 管理画面 + EC の要件をすべてカバー）<br>・商品ページの SEO と管理画面の SPA 的操作を両立 |
| **Remix** | React ベース。Web 標準（Fetch API / FormData）を重視した FW | ・フォーム処理が直感的<br>・ネストルーティングが強力<br>・Progressive Enhancement | ・Next.js に比べエコシステム・情報量が少ない<br>・大規模チームでの採用実績が限定的<br>・React Router v7 との統合移行中 | ・Web 標準重視のチーム<br>・データ更新が多いアプリ（管理画面特化）<br>・25億規模では Next.js のほうがリスクが低い |
| **Nuxt.js（Vue.js）** | Vue.js ベースのフルスタック FW。HTML テンプレートベース | ・Vue.js の直感的な記法<br>・テンプレート構文が HTML に近くデザイナーと協業しやすい<br>・Auto Import が便利 | ・TypeScript サポートが React エコシステムより弱い<br>・大規模 PJ での実績が Next.js に比べ少ない<br>・React 系人材のほうが確保しやすい | ・Vue.js に習熟したチームが確保できる場合<br>・本PJでは技術スタック統一（React → Kotlin）の観点で不採用 |
| **SvelteKit** | Svelte ベース。仮想 DOM を使わずコンパイル時に最適化 | ・バンドルサイズが最小<br>・記述量が少ない<br>・パフォーマンスが高い | ・エコシステム・人材が限定的<br>・大規模開発の実績が少ない<br>・日本語情報が少ない | ・小中規模のプロジェクト<br>・パフォーマンス最優先の場合<br>・25億規模の PJ では人材リスクが高すぎる |

## 4. レンダリング方式

> 詳細は [frontend-mobile-architecture.md](frontend-mobile-architecture.md) セクション1 を参照。

| パターン | 役割 | メリット | デメリット | 判断基準 |
|---|---|---|---|---|
| **SSR + RSC（App Router）** | サーバーで HTML 生成 + React Server Components でクライアント/サーバーを分離 | ・SEO に強い + 初期表示が速い<br>・バンドルサイズ削減<br>・Streaming SSR で部分的に即表示 | ・Server/Client Component 境界の学習コスト<br>・サードパーティ対応が不完全な場合あり | ・**本PJ推奨**（商品ページ SEO + 管理画面 SPA） |
| **SPA（CSR のみ）** | ブラウザで JS が全画面を描画 | ・UX が滑らか<br>・FE/BE 完全分離 | ・SEO が弱い<br>・初期表示が遅い | ・管理画面・認証後ダッシュボードのみ |
| **SSG + ISR** | ビルド時に HTML 生成 + 一定時間で再生成 | ・表示最速（CDN直配信）<br>・サーバーコスト最小 | ・リアルタイム性が低い<br>・在庫・価格など動的データに不向き | ・商品一覧（数分遅れ OK なページ）に部分適用 |

## 5. スタイル設計

| パターン | 役割（CSS をどう書くか） | メリット | デメリット | 判断基準 |
|---|---|---|---|---|
| **Tailwind CSS** | ユーティリティクラスを HTML に直接記述。`className="flex gap-4 p-2"` のようにスタイルを当てる | ・書く速度が速い<br>・デザインシステムとの一貫性（spacing / color がトークンで統一）<br>・バンドルサイズが小さい（未使用を自動削除）<br>・RSC（Server Component）と完全互換 | ・HTML が長くなる（`className` が肥大化）<br>・CSS の知識は依然として必要<br>・チーム全員に Tailwind の記法を覚えてもらう必要あり | ・**本PJ推奨**（App Router + RSC と相性最良）<br>・デザインシステムをゼロから構築する場合<br>・コンポーネント単位の開発に向いている |
| **CSS Modules** | `.module.css` ファイルにスコープ付き CSS を記述 | ・CSS の標準記法がそのまま使える<br>・スコープが自動でモジュール化（名前衝突なし）<br>・Next.js 公式サポート<br>・RSC と互換性あり | ・ファイル数が増える（.tsx + .module.css）<br>・動的スタイルの記述がやや冗長<br>・デザイントークンの一元管理は自分で仕組みを作る必要あり | ・CSS 経験が豊富なチーム<br>・既存の CSS 資産を活かしたい場合<br>・Tailwind を覚えるコストを避けたい場合 |
| **Styled Components / Emotion（CSS-in-JS）** | JavaScript の中に CSS を書く。`styled.div` や `css` 関数で記述 | ・動的スタイルが直感的<br>・コンポーネントとスタイルが同一ファイル<br>・テーマ機能が組み込み | ・**RSC（Server Component）と非互換**<br>・ランタイムの JS オーバーヘッド<br>・バンドルサイズが増える | ・**App Router 環境では非推奨**<br>・Pages Router のレガシー PJ の保守には使える<br>・新規開発では Tailwind or CSS Modules を推奨 |
| **Vanilla Extract** | TypeScript で型安全に CSS を書く。ビルド時に静的 CSS を生成 | ・型安全（定義漏れをコンパイルエラーで検出）<br>・ゼロランタイム（CSS-in-JS の欠点がない）<br>・テーマのトークン管理が強力 | ・記法の学習コストがやや高い<br>・日本語の情報が少ない<br>・エコシステムが Tailwind ほど大きくない | ・デザインシステムを厳格に型で管理したい場合<br>・CSS-in-JS が好きだが RSC 対応が必要な場合 |
| **shadcn/ui + Tailwind** | Tailwind ベースの UI コンポーネント集。`npx shadcn-ui add button` でソースコードごとコピー | ・コンポーネントのソースコードを所有できる（ロックインなし）<br>・Tailwind との統合が完璧<br>・アクセシビリティ対応済み（Radix UI ベース） | ・コンポーネントをコピーするのでコード量が増える<br>・アップデートは手動での差分反映<br>・大規模カスタマイズは自前の設計が必要 | ・**本PJ推奨**（Tailwind 採用時の UI ベースとして）<br>・EC に必要なコンポーネント（Button / Dialog / Table / Form）を素早く構築 |

## 6. API 通信ライブラリ

| パターン | 役割（API をどう呼ぶか） | メリット | デメリット | 判断基準 |
|---|---|---|---|---|
| **fetch API（ネイティブ）** | ブラウザ / Node.js 組み込みの HTTP クライアント。追加ライブラリ不要 | ・依存ゼロ<br>・Next.js の App Router で拡張済み（キャッシュ / revalidate 対応）<br>・Server Component で直接使える | ・インターセプター（認証ヘッダー自動付与等）が自前実装<br>・エラーハンドリングが冗長（res.ok チェック必要）<br>・リクエストのキャンセルは AbortController が必要 | ・**RSC（Server Component）側の API 呼び出しはこれ一択**<br>・Next.js のキャッシュ機能をフル活用したい場合 |
| **Axios** | HTTP クライアントライブラリ。インターセプターやリクエスト変換が豊富 | ・インターセプターで認証トークン自動付与が容易<br>・リクエスト/レスポンス変換が便利<br>・広く普及しており情報が豊富 | ・バンドルサイズが fetch より大きい（~13KB gzip）<br>・fetch + TanStack Query と役割が被る<br>・Node.js 側は fetch に標準化の流れ | ・Client Component 側でインターセプターが必要な場合<br>・チームが Axios に慣れている場合 |
| **ky** | fetch のラッパー。軽量でモダンな API デザイン | ・fetch ベースで軽量（~3KB）<br>・リトライ内蔵<br>・JSON 自動パース | ・Axios ほどの知名度がない<br>・インターセプターは hook 形式（Axios と異なる） | ・fetch の DX を上げつつ軽量に保ちたい場合 |
| **openapi-fetch** | OpenAPI 定義から型安全に API を呼ぶ。fetch ベース | ・**OpenAPI 定義ファーストと相性最良**<br>・型安全（エンドポイント + リクエスト + レスポンスすべて型付き）<br>・fetch ベースで軽量 | ・OpenAPI 定義がないと使えない<br>・エコシステムが Axios ほど大きくない | ・**本PJ推奨**（OpenAPI 定義ファーストの方針と一致）<br>・型安全で FE/BE の API 不整合を防ぐ |

## 8. テスト戦略

| パターン | 役割（何をテストするか） | メリット | デメリット | 判断基準 |
|---|---|---|---|---|
| **Unit Test（ユニットテスト）** | 個々の関数・フックが正しく動くかを検証 | ・実行が高速（ミリ秒単位）<br>・バグの原因特定が容易<br>・リファクタリングの安全網 | ・UI の見た目は検証できない<br>・実装に密結合すると壊れやすいテストになる | ・ビジネスロジック（価格計算・在庫判定等）は**必須**<br>・ユーティリティ関数は費用対効果が高い |
| **Integration Test（結合テスト）** | 複数のコンポーネントが連携して動くかを検証。API モックを使用 | ・ユーザー操作の流れをテストできる<br>・コンポーネント間の連携バグを検出<br>・Unit Test より現実的 | ・セットアップがやや複雑<br>・実行速度が Unit Test より遅い | ・**本PJ推奨**（主要な画面フローにカバレッジを確保）<br>・注文フロー・カート操作など重要な業務フロー |
| **E2E Test（エンドツーエンドテスト）** | ブラウザを実際に操作し、FE→BE→DB まで通しで検証 | ・最もユーザーに近いテスト<br>・システム全体の連携を検証<br>・リグレッション防止に最も効果的 | ・実行が遅い（秒〜分単位）<br>・環境依存でフレイキーになりやすい<br>・メンテナンスコストが高い | ・重要業務フロー（商品検索→カート→決済→注文完了）に限定適用<br>・全画面に適用するのはコスト過大 |
| **Snapshot Test** | コンポーネントの描画結果（HTML）を保存し、変更を検出 | ・UI の意図しない変更を検出<br>・セットアップが簡単 | ・「更新して終わり」になりがちで形骸化しやすい<br>・なぜ変わったかの判断が困難 | ・主要な共通コンポーネント（ヘッダー・フッター等）に限定<br>・やらないという選択肢も合理的 |
| **Visual Regression Test** | スクリーンショットを比較し、ピクセル単位の差分を検出 | ・CSS の意図しない変更を視覚的に検出<br>・Snapshot Test より正確 | ・ツール導入コスト（Chromatic / Percy / reg-suit 等）<br>・CI 時間が増える<br>・微小な差分（フォント等）で偽陽性 | ・デザインの品質基準が厳格な場合<br>・共通 UI コンポーネントライブラリの保守に効果的 |

### テスト比率の推奨（テストピラミッド）

```
        ╱╲
       ╱E2E╲         ← 少なく（重要フローのみ 5-10本）
      ╱──────╲
     ╱Integration╲    ← 中程度（主要画面フロー 50-100本）
    ╱──────────────╲
   ╱   Unit Test    ╲  ← 多く（ロジック中心 200本以上）
  ╱──────────────────╲
```

## 15. BFF（Backend for Frontend）

### 15.1 そもそも BFF とは何か

```
【BFF なし】
  ブラウザ ──→ Spring Boot API（:8080）
                直接やりとり

【BFF あり】
  ブラウザ ──→ BFF（:3000）──→ Spring Boot API（:8080）
                ↑ 間に入る中間層
                ここで API の集約・変換・認証トークン管理を行う
```

BFF は「フロントエンド専用の裏方サーバー」です。  
ユーザーには見えませんが、FE と BE の間に立って以下を担当します:

1. **API の集約**: 1画面に必要な複数 API を BFF で1回にまとめる
2. **レスポンスの変換**: BE のレスポンスを FE に最適な形に加工
3. **認証トークンの管理**: アクセストークンを Server 側で保持（ブラウザに見せない）
4. **CORS の解消**: 同一オリジンになるので CORS 設定が不要

---

### 15.2 パターン比較

| パターン | 役割（FE と BE の間に中間層を置くか） | メリット | デメリット | 判断基準 |
|---|---|---|---|---|
| **Next.js Route Handlers を BFF として使う** | Next.js 内の API エンドポイント（`app/api/`）で BE API の集約・変換を行う | ・追加インフラ不要（Next.js に内蔵）<br>・FE チームが自分で API を加工できる<br>・CORS 問題の回避<br>・認証トークンの管理を Server 側で行える | ・Next.js の役割が増える（FE + BFF）<br>・パフォーマンスのボトルネックになりうる<br>・BE API と BFF API の二重管理 | ・**本PJ推奨**（軽量な BFF は Route Handlers で十分）<br>・認証トークンの Cookie 管理に最適 |
| **専用 BFF サーバー（Node.js / Express / Fastify 等）** | Next.js とは別プロセスで専用の BFF を立てる | ・BFF が独立してスケール可能<br>・Next.js のバージョンに依存しない<br>・複雑な集約ロジックを収容しやすい | ・インフラ・運用コストが増える<br>・デプロイパイプラインが別途必要<br>・チーム間の責任境界が曖昧になりうる | ・BFF のロジックが複雑で Next.js に収まらない場合<br>・マイクロサービス構成で BFF を独立させたい場合 |
| **BFF なし（FE → BE 直結）** | フロントエンドから Spring Boot の API を直接呼ぶ | ・シンプル（中間層がない）<br>・デバッグが容易<br>・レイテンシが最小 | ・FE の都合で BE API を変更する必要が出る<br>・CORS 設定が必要<br>・認証トークンのクライアント管理が複雑 | ・BE API が完全に FE の要件を満たしている場合<br>・BFF 層のメリットが不要な小規模 PJ |

#### ① 認証トークンをブラウザに見せない（セキュリティ最大のメリット）

```

  ブラウザ                              Spring Boot
  ┌─────────────────────┐              ┌──────────┐

  │ メモリに             │ ──────────→  │  API     │

  └─────────────────────┘              └──────────┘
       ↑
       XSS 攻撃でトークンが盗まれるリスク
       → 攻撃者がユーザーになりすまして API を呼べてしまう

【BFF あり — トークンは Server 側だけが知っている】

  ブラウザ                   BFF（Next.js）            Spring Boot
  ┌──────────┐  HttpOnly    ┌──────────────┐  Bearer   ┌──────────┐
  │ Cookie   │  Cookie      │ セッション    │  token    │          │
  │ (session │ ──────────→ │ ↔ トークン   │ ───────→ │  API     │
  │  IDのみ)  │             │ サーバー側で  │           │          │
  └──────────┘             │ 保持         │           └──────────┘
       ↑                    └──────────────┘
       XSS でも Cookie の中身は読めない（HttpOnly）
       → トークンが盗まれない  ✅
```

**本PJでの重要性**: 海外販売 EC では決済・個人情報を扱うため、トークン管理のセキュリティは必須要件。

---

#### ② API の集約 — 1画面に必要な複数 API をまとめる

```
【BFF なし — ブラウザから複数 API を個別に呼ぶ】

  ブラウザ（商品詳細画面）
    ├→ GET /api/products/123          （商品情報）
    ├→ GET /api/products/123/reviews  （レビュー）
    ├→ GET /api/inventory/123         （在庫状況）
    ├→ GET /api/products/123/related  （関連商品）
    └→ GET /api/pricing/123?currency=USD （価格）

    → 5回の HTTP リクエスト
    → モバイル回線では遅い（RTT × 5）
    → 1つ失敗したときのエラーハンドリングが複雑

【BFF あり — 1回のリクエストで全部取得】

  ブラウザ
    └→ GET /api/bff/products/123  （← この1回だけ）

  BFF 内部で:
    const [product, reviews, inventory, related, pricing] =
      await Promise.all([
        fetch("http://backend:8080/api/products/123"),
        fetch("http://backend:8080/api/products/123/reviews"),
        fetch("http://backend:8080/api/inventory/123"),
        fetch("http://backend:8080/api/products/123/related"),
        fetch("http://backend:8080/api/pricing/123?currency=USD"),
      ]);

    → ブラウザ ↔ BFF は 1回の通信
    → BFF ↔ BE は同一ネットワーク内なのでレイテンシが極小
    → ブラウザではなくサーバー側でまとめるので高速  ✅
```

#### ③ CORS を解消

```
【BFF なし — オリジンが異なる】

  https://shop.example.com（FE: Next.js）
    → https://api.example.com（BE: Spring Boot）
    → オリジンが異なる = CORS 設定が必要
    → BE 側の SecurityConfig.java に設定追加が必要
    → プリフライトリクエスト（OPTIONS）が毎回飛ぶ（パフォーマンス低下）

【BFF あり — 同一オリジン】

  https://shop.example.com/api/bff/products  （BFF = Next.js Route Handler）
  https://shop.example.com/products          （FE = Next.js ページ）
    → 同じオリジン = CORS の問題なし  ✅
    → プリフライトリクエストも不要
```

#### ④ レスポンスの変換 — FE に最適な形に加工

```
【BE のレスポンス（Spring Boot 側の都合）】
{
  "productId": 123,
  "productName": "KALLAX Shelf Unit",
  "priceInMinorUnits": 799000,     ← 通貨の最小単位（分 / 銭）
  "currencyCode": "JPY",
  "stockQuantity": 42,
  "warehouseCode": "WH-003",       ← FE には不要
  "internalCategoryId": "CAT-789", ← FE には不要
  "lastUpdatedBy": "admin",        ← FE には不要
}

【BFF で変換後（FE にとって使いやすい形）】
{
  "id": 123,
  "name": "KALLAX Shelf Unit",
  "price": "¥7,990",              ← 表示用にフォーマット済み
  "priceRaw": 7990,               ← 計算用
  "inStock": true,                ← boolean に変換
  "stockLabel": "在庫あり（残り42点）"
}

→ FE は受け取ったデータをそのまま表示すればOK
→ 価格フォーマット・在庫判定などのロジックが BFF に集約される  ✅
```

#### ⑤ FE と BE のデプロイを独立できる

```
【BFF がクッションになる】

  BE が API のレスポンス構造を変更した
    ↓
  BFF の変換ロジックを修正するだけ
    ↓
  FE は変更不要（BFF のレスポンスは変わらないから）

  → FE と BE のデプロイタイミングを合わせなくてよい  ✅
  → 25億規模の PJ ではチーム間のデプロイ調整コストが大きいので重要
```

### 15.4 デメリット詳細

#### ① レイテンシが増える（ネットワークホップが1段増える）

```
【BFF なし】
  ブラウザ ──→ Spring Boot     = 1ホップ（例: 50ms）

【BFF あり】
  ブラウザ ──→ BFF ──→ Spring Boot  = 2ホップ（例: 50ms + 5ms = 55ms）
                 ↑
                 BFF → BE は同一ネットワーク内なので追加は数ms程度
                 → API 集約の恩恵のほうが大きい場合が多い

  ただし注意:
  ・BFF で重い処理（画像変換・大量データ変換等）をやると遅くなる
  ・BFF は「軽い集約・変換」に留めるべき
  ・ビジネスロジックは BFF に書かない（それは BE の仕事）
```

---

#### ② 障害ポイントが増える（BFF が落ちると全部止まる）

```
【BFF なし】
  Spring Boot が落ちたら → 止まる（1箇所だけ気にすればいい）

【BFF あり】
  BFF（Next.js）が落ちたら → 全画面止まる
  Spring Boot が落ちたら → → 全画面止まる
  → 障害ポイントが2箇所に増える

  対策:
  ・Next.js のヘルスチェック → ALB で異常検知 → 自動復旧
  ・BFF のエラーハンドリング → BE が503でもFEがフォールバック表示
  ・可能なら BFF を複数台で冗長構成にする
```

---

#### ③ 開発・運用コストが増える

```
【追加される作業】

  開発:
  ・Route Handlers の実装（BFF のエンドポイント）
  ・BE のレスポンス → FE 用レスポンスの変換ロジック
  ・BFF 用のテスト

  運用:
  ・BFF のログ監視
  ・BFF のパフォーマンスモニタリング
  ・BE API の仕様変更時に BFF も追随

  ただし:
  ・Next.js の Route Handlers を使えばインフラの追加コストはゼロ
  ・専用 BFF サーバーを別に立てるのとは負荷が大きく異なる
  ・「Next.js の Route Handlers」≒ ほぼ追加コストなし
  ・「専用 Node.js サーバー」≒ インフラ + デプロイ + 監視が別途必要
```

---

#### ④ BFF にビジネスロジックが漏れ出すリスク

```
【やってはいけない例】

  // BFF（Route Handler）内で...
  if (order.total > 10000) {
    // 10,000円以上なら送料無料にする ← これはビジネスロジック！
    order.shippingFee = 0;
  }

  → この判定は Spring Boot のドメイン層でやるべき
  → BFF がビジネスルールを持つと「どこに何があるか」が不明確になる
  → バグの温床になる

【BFF でやっていいこと】
  ✅ 複数 API の集約（Promise.all で並列呼び出し）
  ✅ レスポンスの形式変換（snake_case → camelCase、価格フォーマット等）
  ✅ 認証トークンの管理（Cookie ↔ Bearer Token の変換）
  ✅ エラーレスポンスの統一フォーマット変換

【BFF でやってはいけないこと】
  ❌ 価格計算・割引ロジック
  ❌ 在庫の引当判定
  ❌ 注文ステータスの遷移制御
  ❌ データの永続化（DB 接続）
```

---

#### ⑤ デバッグが複雑になる

```
【BFF なし — デバッグが直線的】

  ブラウザの DevTools → Network タブ → Spring Boot のレスポンスが見える
  → 問題の切り分けが単純

【BFF あり — 「どこが原因か」の切り分けが必要】

  ブラウザ → BFF → Spring Boot

  バグが出たとき:
  ① ブラウザの DevTools で BFF のレスポンスを確認
  ② BFF のログで Spring Boot から何が返ってきたか確認
  ③ Spring Boot のログで内部の処理を確認
  → 3箇所を見る必要がある

  対策:
  ・リクエスト ID（X-Request-Id）を全層で受け渡す → ログを横串で追える
  ・BFF のログに「BE からのレスポンス」「FE に返すレスポンス」の両方を出力
```

### 15.6 コード例（Next.js Route Handler での BFF）

```typescript
// app/api/bff/products/[id]/route.ts
import { cookies } from "next/headers";

export async function GET(
  request: Request,
  { params }: { params: { id: string } }
) {
  // ① Cookie からセッションを取得 → アクセストークンに変換
  const sessionId = cookies().get("session_id")?.value;
  const accessToken = await resolveAccessToken(sessionId);

  // ② 複数の BE API を並列で呼ぶ
  const [product, reviews, inventory] = await Promise.all([
    fetch(`${process.env.BACKEND_URL}/api/products/${params.id}`, {
      headers: { Authorization: `Bearer ${accessToken}` },
    }).then(res => res.json()),

    fetch(`${process.env.BACKEND_URL}/api/products/${params.id}/reviews`, {
      headers: { Authorization: `Bearer ${accessToken}` },
    }).then(res => res.json()),

    fetch(`${process.env.BACKEND_URL}/api/inventory/${params.id}`, {
      headers: { Authorization: `Bearer ${accessToken}` },
    }).then(res => res.json()),
  ]);

  // ③ FE に最適な形に変換して返す
  return Response.json({
    id: product.productId,
    name: product.productName,
    price: formatCurrency(product.priceInMinorUnits, product.currencyCode),
    priceRaw: product.priceInMinorUnits / 100,
    inStock: inventory.stockQuantity > 0,
    stockLabel: inventory.stockQuantity > 0
      ? `在庫あり（残り${inventory.stockQuantity}点）`
      : "在庫なし",
    reviews: reviews.items.map(r => ({
      author: r.userName,
      rating: r.score,
      comment: r.body,
    })),
  });
}
```

### 15.7 判断フローチャート

```
BFF を導入すべきか？

  ① 認証トークンをブラウザに見せたくない？
     → YES → BFF 推奨（HttpOnly Cookie でトークン管理）

  ② 1画面で複数 API を呼んでいる？
     → YES → BFF で集約するとパフォーマンス向上

  ③ BE の API レスポンスが FE にとって使いにくい？
     → YES → BFF で変換するとFE の実装がシンプルに

  ④ CORS の設定が面倒 / セキュリティ要件が厳しい？
     → YES → BFF（同一オリジン）で解消

  上記のいずれかに YES → BFF 導入を推奨（本PJは①②③すべて該当）
  すべて NO → BFF なしで直結がシンプル
```

## 16. 認証方式

> ⚠️ 認証は想定の 1.5〜2倍の工数がかかる。必ずバッファを取ること。

| パターン | 役割（ユーザーをどう認証するか） | メリット | デメリット | 判断基準 |
|---|---|---|---|---|
| **Keycloak + OIDC（Authorization Code Flow + PKCE）** | Keycloak をアイデンティティプロバイダーとし、OIDC 準拠で認証。FE は PKCE 付きの認証コードフローを使用 | ・業界標準の OIDC/OAuth2 プロトコル<br>・多言語対応・SSO がKeycloakに組み込み<br>・ロール管理・権限管理が Keycloak の管理画面で完結<br>・認証ロジックを自前実装しなくてよい | ・Keycloak の運用知識が必要<br>・トークンリフレッシュ処理の実装が必要<br>・Keycloak がSPOFになるリスク（HA構成必要） | ・**本PJ確定**（Keycloak 採用済み）<br>・多言語 EC + 管理画面の認証に最適 |
| **NextAuth.js (Auth.js)** | Next.js 向けの認証ライブラリ。各種プロバイダー対応 | ・Next.js との統合が容易<br>・セッション管理が組み込み<br>・多数のプロバイダー対応 | ・Keycloak の機能と重複する部分がある<br>・Auth.js v5 は移行期で破壊的変更あり | ・Keycloak を使わない場合の選択肢<br>・ソーシャルログインのみで良い小規模案件 |
| **Cookie ベースセッション（BFF 経由）** | BFF（Route Handlers）でトークンを管理し、FE には HttpOnly Cookie でセッション ID を渡す | ・アクセストークンがブラウザの JS から見えない（XSS 対策）<br>・FE 側の認証ロジックがシンプル | ・BFF 層の実装が必要<br>・サーバー側セッション管理のスケーラビリティ | ・**本PJ推奨の認証トークン管理方式**（Keycloak + BFF Cookie） |

### 認証で気をつけるべきこと

```
【リフレッシュトークンのフロー】

  ① アクセストークン（短命: 5分〜15分）で API を呼ぶ
  ② 401 Unauthorized が返ってきた
  ③ リフレッシュトークン（long-lived）で新しいアクセストークンを獲得
  ④ 元の API リクエストを再送
  ⑤ リフレッシュトークンも切れていたら → ログイン画面にリダイレクト

  → この ②③④ の自動化が意外と大変
  → 同時に複数 API を呼んでいる場合、リフレッシュの「競合」が発生する
  → 1回だけリフレッシュして、待っていたリクエストを一括再送する仕組みが必要
```

## 17. エラーハンドリング方針

| パターン | 役割（エラーメッセージを誰が持つか） | メリット | デメリット | 判断基準 |
|---|---|---|---|---|
| **FE で持つ（エラーコードで分岐）** | BE は `errorCode: "ORDER_001"` 等のコードを返し、FE がユーザー向けメッセージを表示 | ・多言語対応が FE 側で完結<br>・ユーザー向けの文言を FE チームがコントロール<br>・BE の API メッセージが技術的でも問題なし | ・エラーコードの定義・管理が必要<br>・FE が全エラーコードを知っている必要がある<br>・新しいエラーコード追加時に FE 修正が必要 | ・**本PJ推奨**（多言語対応 + ユーザー体験の最適化）<br>・EC サイトではユーザー向けメッセージの品質が重要 |
| **BE のメッセージをそのまま表示** | BE が `message: "在庫が不足しています"` のようにユーザー向けメッセージを返す | ・FE の実装がシンプル<br>・メッセージの一元管理 | ・多言語対応を BE 側でやる必要がある<br>・BE が UI 要件（文言の長さ・トーン等）を考慮する必要がある<br>・API 利用者向けの技術メッセージと混在するリスク | ・単一言語の小規模アプリ<br>・BE がユーザー向けメッセージを返す設計の場合 |
| **ハイブリッド（既知は FE / 未知は BE）** | 想定内のバリデーションエラーは FE、500 等の想定外エラーは BE の汎用メッセージを表示 | ・バランスが良い<br>・想定外のエラーも最低限のメッセージが出る | ・「想定内」と「想定外」の境界の定義が必要 | ・多くの PJ で現実的な落とし所 |

## 18. メンテナンスモード

| パターン | 役割（メンテナンス中をどう表示するか） | メリット | デメリット | 判断基準 |
|---|---|---|---|---|
| **API レスポンス + FE で検出** | BE が 503 + 特定の JSON を返し、FE のグローバルインターセプターで検出してメンテ画面に切り替え | ・SPA 遷移中でも検出できる（クライアントルーティング中の対応可能）<br>・メンテ中/解除のタイミングが柔軟 | ・FE 側にインターセプターの実装が必要<br>・メンテモードのテストが確認しづらい | ・**本PJ推奨**（SPA のクライアントルーティング中も対応可） |

## 19. バージョン管理・ブランチモデル

### 19.1 パターン比較

| パターン | 役割（ブランチをどう管理するか） | メリット | デメリット | 判断基準 |
|---|---|---|---|---|
| **GitHub Flow（原型）** | `main` + feature ブランチ。PR マージで本番にデプロイ | ・シンプルで理解しやすい<br>・CI/CD と相性が良い<br>・小回りが利く | ・リリースタイミングの制御が弱い<br>・ステージング検証のフローが自前設計必要 | ・継続的デプロイが可能な環境<br>・FE 単体の小規模プロダクト |
| **Git Flow** | `main` + `develop` + feature + release + hotfix の5種ブランチ | ・リリースタイミングを厳密にコントロール<br>・ステージング検証の手順が明確<br>・ホットフィックスの手順が定型化 | ・ブランチが多く運用が複雑<br>・マージ頻度が高いとコンフリクトが増える<br>・CI/CD パイプラインが複雑化 | ・リリースサイクルが決まっている場合（月次リリース等）<br>・BE と FE のリリースを同期させる必要がある場合 |
| **Trunk Based Development** | `main` に直接（または短命ブランチで）マージ。Feature Flag で機能制御 | ・コンフリクトが最小化<br>・CI/CD が最速<br>・デプロイ頻度を最大化できる | ・Feature Flag の設計・管理が必要<br>・チーム全員に高い技術力が必要<br>・未完成機能が main に入るリスクの管理 | ・技術力が高いチーム<br>・デプロイを毎日 or 週次で回したい場合 |
| **GitHub Flow + Environment Branch（dev / stg）** | `main` + `stg` + `dev` + feature。環境ブランチで段階的に昇格 | ・GitHub Flow のシンプルさがベース<br>・環境ごとの CI/CD パイプラインが明確<br>・dev で統合テスト → stg で受入テスト → main で本番の3段階品質ゲート<br>・ドキュメントは main 直結で運用可能 | ・dev ↔ stg ↔ main の差分管理が必要<br>・純粋な GitHub Flow よりブランチが増える | ・**本PJ推奨**<br>・EC サイトではリリース前に stg で検証が必須<br>・BE と FE のデプロイタイミングを合わせやすい |

### 19.2 本PJ推奨: GitHub Flow + Environment Branch（dev / stg）

GitHub Flow をベースに、環境ブランチ（`dev`, `stg`）を追加した構成。

```
【ブランチ構成】

  main（本番） ← stg からの PR マージでデプロイ
    ↑
  stg（検証）  ← dev からの PR マージでデプロイ
    ↑
  dev（開発）  ← feature/* からの PR マージでデプロイ
    ↑
  feature/*    ← dev から切って作業（機能開発）
  hotfix/*     ← main から切って緊急修正

  ※ docs/* は main に直接 PR 可（後述）
```

---

### 19.3 ブランチの役割と対応環境

| ブランチ | 環境 | 役割 | デプロイトリガー |
|---|---|---|---|
| `main` | 本番 | リリース済みコード。常にデプロイ可能な状態を維持 | `stg` → `main` への PR マージ |
| `stg` | ステージング | 受入テスト（UAT）・リリース前の最終検証 | `dev` → `stg` への PR マージ |
| `dev` | 開発 | 機能統合・結合テスト。feature の集約先 | `feature/*` → `dev` への PR マージ |
| `feature/*` | ローカル / プレビュー | 個別機能の開発。短命ブランチ（数日〜1週間） | — |
| `hotfix/*` | — | 本番の緊急修正。`main` から切って `main` に直接マージ | `hotfix/*` → `main` → `stg`, `dev` に逆流 |

### 19.4 通常の開発フロー

```
① feature ブランチを dev から作成
   git switch dev
   git pull origin dev
   git switch -c feature/product-detail-page

② 開発 → コミット → プッシュ
   git add .
   git commit -m "feat: 商品詳細画面を実装"
   git push origin feature/product-detail-page

③ dev への PR を作成
   feature/product-detail-page → dev
   ・CI（lint, test, build）が自動実行
   ・レビュアー1名の Approve で マージ
   ・dev 環境に自動デプロイ → 結合テスト

④ dev で統合テスト完了後、stg への PR を作成
   dev → stg
   ・stg 環境に自動デプロイ
   ・QA チーム / PO が受入テスト（UAT）

⑤ stg で検証完了後、main への PR を作成
   stg → main
   ・リーダーの Approve 必須
   ・本番環境に自動デプロイ（or 承認後デプロイ）
```

### 19.5 ドキュメントの扱い — main 直結ルート

```
【ドキュメント変更は dev / stg を経由しなくてよい】

  docs/* ブランチ → main への直接 PR

  理由:
  ・ドキュメントはアプリの動作に影響しない
  ・検証環境での確認が不要
  ・dev / stg を経由すると無駄にリードタイムが伸びる
  ・素早く情報を共有できることのほうが重要

  運用ルール:
  ・対象: docs/ 配下のファイル、README.md、CHANGELOG.md 等
  ・CI: markdownlint のみ実行（ビルド・テストは不要）
  ・レビュー: 1名 Approve でマージ（内容の妥当性確認のみ）
```

**GitHub Actions での制御例**:

```yaml
# .github/workflows/deploy-production.yml
on:
  push:
    branches: [main]

jobs:
  deploy:
    # docs/ のみの変更ならデプロイをスキップ
    if: |
      github.event_name == 'push' &&
      !contains(github.event.head_commit.message, '[docs-only]')
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      # ... デプロイ処理
```

```yaml
# .github/workflows/docs-lint.yml
on:
  pull_request:
    paths:
      - 'docs/**'
      - '*.md'

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npx markdownlint-cli2 "docs/**/*.md"
```

### 19.6 hotfix の緊急修正フロー

```
【本番で障害が発生した場合】

  ① main から hotfix ブランチを作成
     git switch main
     git switch -c hotfix/fix-payment-error

  ② 修正 → テスト → main に PR
     hotfix/fix-payment-error → main
     ・リーダーの Approve 必須
     ・本番デプロイ

  ③ main の修正を stg, dev に逆流させる
     git switch stg  ; git merge main
     git switch dev  ; git merge main

     → stg, dev に修正が反映されないと次のリリースで退行する
```

### 19.8 全体フロー図

```

                         │    main（本番環境）       │

                         └──────────┬──────────────┘
                                    ↑
              ┌─────────────────────┤
              │                     │
   hotfix/* ──┘          ┌──────────┴──────────────┐
   (main から切る)        │    stg（検証環境）        │
   (main に直接 PR)       │  UAT / 受入テスト        │
                         └──────────┬──────────────┘
                                    ↑
                         ┌──────────┴──────────────┐
                         │    dev（開発環境）        │
             docs/* ─────│→ 統合テスト              │──→ main へ直接 PR
             (main直結)   └──────────┬──────────────┘     (ドキュメントのみ)
                                    ↑
                    ┌───────────────┼───────────────┐
                    │               │               │
              feature/A       feature/B       feature/C
              (dev から切る)   (dev から切る)   (dev から切る)
```

### 19.9 なぜ純粋な GitHub Flow ではなくこの構成か

```
【純粋な GitHub Flow の場合】

  main ← feature/*
  → feature を main にマージ＝即本番デプロイ
  → EC サイトで「テストなしに本番」はリスクが高い

  問題:
  ・決済処理のバグが本番に出たら → 売上損失 + 顧客信頼失墜
  ・在庫連動のバグが本番に出たら → 過剰受注 + オペレーション混乱
  ・25億規模の海外 EC で「即本番」は品質ゲートが足りない

【本PJ構成のメリット】

  dev → stg → main の3段階

  ① dev: 開発者同士の統合テスト
     「自分の feature が他の feature と競合しないか」を確認
     → 日次〜数日サイクル

  ② stg: QA / PO による受入テスト
     「業務フローが正しく動くか」を確認
     本番と同じデータ・同じ設定で検証
     → スプリント末〜リリース前

  ③ main: 本番デプロイ
     stg で検証済み = 安心してリリース
     → リリースサイクルに合わせて（週次 or 隔週）

  しかも:
  ・Git Flow のように release ブランチは不要（シンプルさを維持）
  ・GitHub Flow の「PR ベースの開発」はそのまま
  ・ドキュメントは main 直結で素早く共有
```

## 20. コードレビュー運用

| パターン | 役割（PR をどうレビューするか） | メリット | デメリット | 判断基準 |
|---|---|---|---|---|
| **必須レビュー（1名 Approve でマージ）** | すべての PR に最低1名のレビュアーの Approve を必須とする | ・知識共有が進む<br>・致命的なバグの混入を防止<br>・コード品質の底上げ | ・レビュー待ちがボトルネックになりうる<br>・レビュアーの負荷が集中する場合がある | ・**本PJ推奨**（リーダー or シニアメンバーの Approve） |
| **必須レビュー（2名 Approve でマージ）** | 2名以上の Approve を必須 | ・品質が最大限に担保<br>・1名のレビュー漏れをもう1名がカバー | ・マージまでの時間が長くなる<br>・レビュアーの負荷が2倍 | ・金融系等の品質基準が極めて厳しいプロダクト<br>・チームに十分なレビュアーがいる場合 |
| **レビュー任意** | レビューなしでマージ可能。必要に応じてレビューを依頼 | ・開発速度が最速<br>・レビュー待ちがない | ・品質が個人の技量に依存する<br>・知識が属人化する | ・プロトタイプ段階のみ<br>・25億規模では非推奨 |

### CI パイプラインで実行すべきチェック

```
PR 作成時（自動実行）:
  ① pnpm install（依存解決）
  ② TypeScript 型チェック（tsc --noEmit）
  ③ ESLint
  ④ Prettier チェック（--check）
  ⑤ Unit Test + Integration Test（Vitest）
  ⑥ Build 成功確認（next build）
  ⑦ （任意）E2E Test（Playwright）
  ⑧ （任意）Bundle Size チェック

main マージ時（自動デプロイ）:
  ① 上記チェックすべて
  ② ステージング環境にデプロイ
  ③ E2E テスト（ステージング環境）
  ④ 承認後 → 本番デプロイ
```

## 32. SPA バージョン不整合対策

| パターン | 役割（古い JS を使い続けるリスクをどう防ぐか） | メリット | デメリット | 判断基準 |
|---|---|---|---|---|
| **バージョンチェック API + 強制リロード** | `/api/version` 等で最新バージョンを返し、不一致なら `window.location.reload()` で強制リロード | ・確実に最新版を使わせられる<br>・実装がシンプル | ・ユーザーが作業中のフォーム入力が消えるリスク<br>・API コール回数が増える | ・**本PJ推奨**（フォーム入力保護と組み合わせ）<br>・EC サイトではカート操作中の強制リロードに注意 |
