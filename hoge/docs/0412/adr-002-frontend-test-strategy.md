# ADR-002: フロントエンドテスト戦略

| 項目 | 内容 |
|---|---|
| ステータス | **承認（Accepted）** |
| 日付 | 2026-04-13 |
| 決定者 | テックリード |

---

## 状況

ADR-001にて、フロントエンドアーキテクチャとして**Feature-Based Architecture（Bulletproof React式）**を採用した。その上でフロントエンドのテスト戦略を確立する必要がある。

本PJの前提を整理すると以下の通りである。

| 区分 | 内容 |
|---|---|
| システム規模 | 25億円規模・複数サブドメイン（在庫・発注・物流・店舗・商品マスタ等）・社内業務システム |
| 画面数 | 多数（複数サブドメイン × 複数画面） |
| フロントの責務 | 表示ルール・UI/UX・**入力バリデーション**（ADR-001に明記） |
| バックエンドの責務 | 業務ロジック（Source of Truth）。DDD + ヘキサゴナルアーキテクチャ |
| 変更頻度 | フロントエンドは高い。バックエンドの業務ロジックは相対的に安定 |
| 開発環境 | AI支援開発（GitHub Copilot等）を全エンジニアが利用する前提 |
| HTTPクライアント | axiosに脆弱性が発見されたため不採用。カスタムfetch wrapperを自作する |

---

## 検討の経緯

本ADRは以下の検討プロセスを経て最終決定に至った。この経緯を残すことで、将来の意思決定者が判断根拠を追跡できるようにする。

### フェーズ1：初期提案

テックリードの初期案は以下の通りであった。

| 対象 | 方針 |
|---|---|
| バックエンド | 古典学派による自動単体テストを実施 |
| フロントエンド | 自動テスト（単体・統合）は実施しない。モンキーテスト・打鍵テストで手動検証 |
| E2E | Playwrightによる自動E2Eテストを実施 |

**根拠**：フロントエンドは変更頻度が高く、自動テストのメンテナンスコストが見合わない。業務ロジックはバックエンドに集中しており、フロントの責務は薄い。

### フェーズ2：第三者レビューによる指摘

第三者レビューで以下の2点が指摘された。

1. **「自動化しない」と「テストしない」の混同リスク** — ADRとして残す文書では役割分担の明示が必要
2. **E2Eのスコープ未定義** — 多数画面・サブドメインで「E2Eをやる」だけでは運用破綻する

手動テスト（打鍵・モンキー）の存在を前提とした場合、初期提案の方向性は妥当と評価された。

### フェーズ3：AI支援開発の考慮による再検討 ← 最終決定のきっかけ

**「AI開発支援ツール（GitHub Copilot等）の普及により、テストコードの生成・修正コストが大幅に低下している」** という観点が加わった。

従来、フロント自動テストを避ける最大の理由は「変更頻度が高く、テスト修正コストが開発速度を阻害する」ことであった。しかし：

- AIはスキーマ変更時のテストコード修正を即座に提案できる
- AIはコンポーネント変更時の統合テスト修正パターンを把握している
- テストコードの初期生成コストもAIにより大幅に減少する

この前提変化により、**自動テストのコスト構造が変わった**。「変更頻度が高い = 自動テストのメンテコストが高い」という前提が成立しにくくなっている。

結果として、**自動単体テスト（純粋ロジック）と自動統合テスト（主要フロー）の導入が費用対効果の観点で合理的**と判断し、最終戦略を確定した。

### フェーズ4：axiosからカスタムfetch wrapperへの変更によるMSW再評価

axiosに脆弱性が発見されたため採用を見送り、**カスタムfetch wrapper（apiClient.ts）を自作する方針**に変更した。

この変更によりMSWの位置づけが根本的に変わった。axiosのインターセプターはライブラリ側の実績ある機能であり「動くこと」を信頼できた。自作wrapperは以下のロジックをチームが自前で実装・保証する必要がある。

| wrapperが担う処理 | テストで確認すべき挙動 |
|---|---|
| 認証ヘッダーの付与 | `Authorization: Bearer {token}` が実際のリクエストに含まれるか |
| 401 → セッション切れ処理 | `logout()` が呼ばれ、UIにセッション切れメッセージが表示されるか |
| 500 → ApiErrorのスロー | エラーが正しく伝播し、UIがエラー状態を表示するか |
| リトライロジック | 失敗後に再試行し、成功した場合UIが正常表示されるか |

`vi.mock()` はapi関数の層でモックするためapiClient.tsが実行されない。これらのロジックをテストするには、**fetchをネットワーク層でインターセプトするMSWが唯一の現実的な手段**となる。

また、この検討を通じてテスト手法の非対称性が明確になった。AI支援により自動テスト（Unit / Integration）のカバレッジを網羅的に高めるコストは下がる一方、モンキーテスト・打鍵テストはAIで代替できず工数が線形にかかる。自動テストの充実度を上げることで、繰り返しの手動テスト工数を削減するという方向性を確定した。

## 課題

1. **E2Eのスコープ定義** — 何を守るためのE2Eかを定義しないと、多数画面・多数サブドメインで運用破綻する
2. **自動テストと手動テストの役割分担の明文化** — 「何を自動化し、何を手動で行うか」の基準を明確にする
3. **ツールスタックの確定** — バックエンド・モバイルと対比した形でフロントエンドのツール構成を確定する

---

## 結論

本PJのフロントエンドテスト戦略は以下の**4層の自動テスト + 2種の手動テスト**で構成する。

### テスト戦略全体像

```
┌──────────────────────────────────────────────────────────────┐
│                     テスト戦略全体像                          │
├─────────────────────────────┬────────────────────────────────┤
│        自動テスト             │        手動テスト               │
├─────────────────────────────┼────────────────────────────────┤
│ [1] Static Analysis          │ [手動1] 打鍵テスト              │
│  TypeScript strict + ESLint  │  機能追加・変更時の動作確認      │
│  （常時・全コード）           │  フォーム入力・画面遷移の確認    │
├─────────────────────────────┼────────────────────────────────┤
│ [2] Unit Tests（Vitest）      │ [手動2] モンキーテスト           │
│  schemas/ utils/ の純粋ロジック│  探索的テスト                  │
│                              │  リリース前の品質確認            │
├─────────────────────────────┤                                │
│ [3] Integration Tests        │                                │
│  Vitest + RTL + MSW          │                                │
│  Feature毎の主要ユーザーフロー  │                                │
├─────────────────────────────┤                                │
│ [4] E2E（Playwright）         │                                │
│  クリティカルパスのみ          │                                │
└─────────────────────────────┴────────────────────────────────┘

バックエンド：古典学派の自動単体テスト（JUnit + Mockito）
モバイル　　：自動単体テスト（JUnit/Kotest + MockK）
```

---

### ツールスタック比較

プロジェクト全体のテストツールを並べることで、フロントエンドの選定に一貫性を持たせる。

| 層 | 言語 | テストランナー | モック/スタブ |
|---|---|---|---|
| バックエンド | Java | JUnit | **Mockito**（クラス・メソッドのモック） |
| モバイル | Kotlin | JUnit / Kotest | **MockK**（Kotlin向けモック） |
| フロントエンド | TypeScript | **Vitest** | `vi.mock()`（モジュール）+ **MSW**（HTTP/API） |

フロントエンドのモックは2役に分かれる点に注意する。
- `vi.mock()` → モジュール・関数のモック（Mockito/MockKの役割に相当）
- **MSW（Mock Service Worker）** → fetchをネットワーク層でインターセプト。カスタムfetch wrapperのロジックテストに不可欠

---

### 各層の詳細

#### [1] Static Analysis（TypeScript + ESLint）

**何を守るか**：型の誤り・コーディング規約・Feature間の不正依存

- TypeScriptの厳格モード（`strict: true`）によりコンパイル時に型不整合を検出する
- `eslint-plugin-boundaries` でFeature間の不正importを静的に禁止する（ADR-001のリスク対策と連動）
- コストがほぼゼロで最も広い範囲をカバーする最優先の品質ゲートである

---

#### [2] Unit Tests（Vitest）

**何を守るか**：`features/*/schemas/` のZodバリデーション・`utils/` の純粋関数の正確さ

```
# 自動化する対象の例
features/inventory/schemas/productSchema.ts  → 商品コード13桁フォーマット・桁数チェック
features/orders/schemas/orderSchema.ts       → 発注数量の範囲・開始日 < 終了日の複合チェック
utils/formatters.ts                          → 表示用フォーマット変換（通貨・日付等）
utils/validators.ts                          → 共通バリデーション関数
```

**自動化する根拠**：

- Zodスキーマはブラウザ・DOMに依存しない純粋関数であり、Vitestで数ミリ秒で実行できる
- ADR-001が定めた「バックエンドのドメインルールの検証コピー」であるため、バックエンド変更時の差異を自動検出する手段として価値が高い
- AI支援ツールによりスキーマ変更時のテスト修正が即座に提案される。メンテナンスコストの懸念は解消されている

**何を書かないか**：Reactコンポーネントの描画内容・CSSクラス・DOMの細部

---

#### [3] Integration Tests（Vitest + React Testing Library + MSW）

**何を守るか**：Feature単位の主要ユーザーフローが、コンポーネント・フック・API呼び出しを通じて正しく動作すること

```
# Feature: inventory のテスト例
- 商品一覧を検索してフィルタ結果が表示される（MSWで一覧APIをモック）
- 商品編集フォームで不正値を入力するとバリデーションエラーが表示される
- 商品編集を保存するとAPIが呼ばれ成功トーストが表示される
- APIエラー時にエラーメッセージが表示される（MSWで500を返す）
```

**MSWを使う理由**：2つの目的がある。

1. **カスタムfetch wrapperのロジックをテストするため**：本PJではaxiosの脆弱性判明によりカスタムfetch wrapperを自作する。wrapperには401→logout・500→ApiError・認証ヘッダー付与等のインターセプター相当のロジックが含まれる。`vi.mock()`はapi関数をモックするためwrapperが実行されないが、MSWはfetchをネットワーク層でインターセプトするためwrapperを通じたテストが可能になる。
2. **バックエンド未完成フェーズの並行開発**：`features/*/api/` のAPIコールをインターセプトし、バックエンドの稼働状態に依存しないフロント単独テストを実現する。

**カバレッジ方針**：Feature毎に以下の3シナリオを最低限カバーする。

| シナリオ種別 | 内容 |
|---|---|
| Happy Path | 正常系：ユーザーが期待通り操作し、期待通りの結果が得られる |
| Validation Error | フォームに不正値を入力した際のバリデーションエラー表示 |
| API Error | APIが失敗した際のエラーハンドリング |

---

#### [4] E2E Tests（Playwright）

**何を守るか**：フロント・バックエンド・DB・認証が統合した状態でクリティカルフローが正しく動くこと

**E2Eで守るスコープ**：

| カテゴリ | テスト内容 |
|---|---|
| 認証フロー | ログイン・ログアウト・セッション切れ |
| 主要業務フロー | サブドメイン毎に代表的なHappy Path 1〜2本 |
| クロスサブドメイン | 複数サブドメインをまたぐ業務フロー（例：発注→在庫反映） |

**E2Eで書かないもの**（他の層で担保）：

- バリデーションエラーの全パターン → Integration Tests で担保
- 純粋ロジックのエッジケース → Unit Tests で担保
- 探索的なUX確認 → 手動テストで担保

**実行方針**：

| 実行タイミング | 対象 | 目的 |
|---|---|---|
| PR時（Smoke） | 認証 + 各サブドメインの代表フロー | マージ前の最低限の品質ゲート |
| 夜間バッチ（Full） | 全E2Eシナリオ | リグレッション検出 |

---

#### 手動テスト（打鍵・モンキー）

自動テストでは補えない以下の観点を担当する。

| テスト種別 | 目的 | タイミング |
|---|---|---|
| **打鍵テスト** | 機能追加・変更後の動作確認、入力バリデーションの体感確認 | 開発完了後・リリース前 |
| **モンキーテスト** | 設計者が予期しない操作パターン・UX上の問題の探索 | リリース前の最終品質確認 |

---

### テスト戦略の役割分担まとめ

| テスト種別 | 自動/手動 | 守るもの | 補完関係 |
|---|:---:|---|---|
| Static Analysis | 自動 | 型・規約・依存の誤り | すべての層の前提 |
| Unit Tests（Vitest） | 自動 | バリデーション・純粋ロジックの正確さ | E2Eのエッジケース負担を軽減 |
| Integration Tests（RTL+MSW） | 自動 | Feature単位のユーザーフロー | E2Eのカバレッジ範囲を絞る根拠 |
| E2E（Playwright） | 自動 | システム結合時のクリティカルフロー | 手動テストのリグレッション負担を軽減 |
| 打鍵テスト | 手動 | 変更後の機能動作・体感UX | 自動テストの「想定内シナリオ」を補完 |
| モンキーテスト | 手動 | 予期しない操作・探索的バグ | 自動テストが設計できない観点 |

## なぜAI支援開発がテスト戦略を変えるのか

従来の「フロント自動テスト避ける」論拠は **「変更頻度 × メンテナンスコスト」** の積が高いことにあった。

```
【従来】
変更頻度（高）× メンテナンスコスト（高）= 総コスト（過剰）→ 自動化しない

【AI支援あり】
変更頻度（高）× メンテナンスコスト（低：AIが修正提案）= 総コスト（許容範囲）→ 自動化する
```

ただしAIがコストを下げるのは**パターン化された修正**に限る。

| AIが得意なこと | AIが苦手なこと |
|---|---|
| スキーマ変更時のテスト修正 | どのシナリオをテストすべきか（設計判断） |
| RTLのクエリ修正 | テストが本当に意図した挙動を検証しているか（品質判断） |
| MSWハンドラーの型合わせ | 新たな業務ルールを正しくテストで表現すること |

**AIはテストの実装コストを下げるが、テストの設計責任はエンジニアが持つ。** 本ADRが定めるカバレッジ方針（Feature毎3シナリオ・描画詳細を書かない等）は、AIに委ねる前にチームで合意した設計基準である。

## テスト戦略の判断基準（ガイドライン）

```
Q1: 型・構造・依存の誤りを検出したいか？
    → 自動：TypeScript strict + ESLint

Q2: バリデーションロジック・純粋関数の正確さを検証したいか？
    → 自動：Vitest 単体テスト

Q3: ユーザーが「操作して結果を得る」フローを検証したいか？
    → 自動：Vitest + RTL + MSW 統合テスト（Feature毎3シナリオ）

Q4: システム全体がつながって動くことを確認したいか？
    → 自動：Playwright E2E（クリティカルパスのみ）

Q5: 変更後の動作・体感UXを確認したいか？
    → 手動：打鍵テスト

Q6: 予期しない操作パターン・UX問題を探索したいか？
    → 手動：モンキーテスト

Q7: UIコンポーネントの描画内容・DOMの細部をテストしたいか？
    → 書かない（変更に弱く、費用対効果が低い）
```

## リスクと対策

| リスク | 対策 |
|---|---|
| Integration TestsがUIの実装詳細に依存して壊れやすくなる | RTLの `getByRole` / `getByLabelText` 等のアクセシビリティクエリを使い、classNameやDOM構造に依存しないテストを書く |
| MSWのモックとバックエンドの実APIに乖離が生じる | バックエンドのOpenAPI仕様（またはTypeScript型定義）からMSWハンドラーを生成し、型の整合性を担保する |
| E2EがFlaky（不安定）になる | `waitForResponse` 等で確定的な待機を行い、固定sleepを排除する。Playwrightのretry設定でCI安定性を担保する |
| AI生成テストの品質が低い | AIが書いたテストコードはレビューを必須とする。「何を検証しているか」が不明なテストはリジェクトする |
| 「描画詳細を書かない」ルールが守られない | コードレビューチェックリストに「DOMのクラス名・構造に依存していないか」を含める |

---

## 採用しなかった選択肢

| 選択肢 | 不採用理由 |
|---|---|
| フロントの全コンポーネントへの単体テスト | 描画詳細のテストはAI支援があっても変更に弱い。統合テストで代替可能 |
| Visual Regression Testing（Storybook + Chromatic等） | 費用・運用コストに対して社内業務システムとしての優先度が低い。デザインシステムが確立した段階での導入を検討する |
| Contract Testing（Pact等） | マルチレポ構成でバックエンドとの乖離リスクはあるが、OpenAPI型定義の共有とMSWの型整合性で代替可能と判断。将来的な採用候補として保留 |

---

## 結論（再掲）

```
【自動テスト】
  [1] Static Analysis  : TypeScript strict + ESLint（常時）
  [2] Unit Tests       : Vitest（schemas/ utils/ の純粋ロジック）
  [3] Integration Tests: Vitest + RTL + MSW（Feature毎3シナリオ）
  [4] E2E              : Playwright（クリティカルパスのみ・Smoke+夜間Full）

【手動テスト】
  打鍵テスト    : 機能変更後の動作確認
  モンキーテスト: 探索的テスト・リリース前品質確認
```

AI支援開発の普及によりテストコードのメンテナンスコストが低下したことで、フロントエンドの自動テストを「純粋ロジック + 主要ユーザーフロー」まで広げることが費用対効果の観点で合理的と判断した。ただしAIはテストの実装コストを下げるに留まり、何をテストするかの設計判断は本ADRが定める基準に従う。
