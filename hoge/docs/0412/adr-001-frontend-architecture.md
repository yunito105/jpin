## 状況

25億円規模の海外販売システムを構築する。業態は小売業（IKEA・ニトリ・カインズに類する）。

本システムはECのような単一巨大アプリではない。在庫管理・発注・物流・店舗管理・商品マスタなど**複数のサブドメインからなる社内業務システム群**であり、画面数も多い。

技術スタックは以下の通り決定済みである。

| 区分 | 技術選定 |
|---|---|
| フロントエンド | React + Vite + TypeScript |
| バックエンド | Java + Spring Boot + Oracle |
| バックエンドアーキテクチャ | ヘキサゴナルアーキテクチャ + 簡易CQRS（DDD導入） |
| リポジトリ戦略 | マルチレポ（フロント・モバイル・バックエンドを分離） |

**責務の分界点**として、業務ロジックの正（Source of Truth）はバックエンド（DDD）に置く。ただしフロントエンドは**表示ルール・UI/UX・入力バリデーション**を責務として持つ。

例えば商品コードの13桁フォーマット変換や、必須チェック・桁数チェックなどの入力バリデーションはフロントエンドで行う。10項目入力後にサーバー送信して初めてエラーが返るようでは、ユーザビリティが著しく損なわれるためである。また、不正な入力を早期に弾くことはセキュリティ対策（Defense in Depth）としても有効である。

つまりフロントエンドは**ドメインルールの「検証コピー」を薄く持つ**。ただし、最終的な整合性保証はバックエンドが担い、フロントのバリデーションは「UXの最適化とセキュリティの第一防衛線」として位置づける。

## Reactの特性とアーキテクチャ選定の必要性

### Reactはライブラリであり、フレームワークではない

バックエンドの Spring Boot（Java）やモバイルの Jetpack（Android）は「フレームワーク」である。アプリケーション全体の構造—ディレクトリ構成・クラスの役割・処理の流れ—がフレームワーク側で規定されている。バックエンドエンジニアが初めて触るコードベースでも「認証処理は Service 層にある」「DB操作は Repository 層にある」と即座に判断できるのは、Spring Boot がその規則を強制しているからである。

**React はUIの描画に特化した「ライブラリ」であり、アプリ全体の構造を規定する機能を持たない。**

> *"React is very flexible, you can write React applications in any way you like, but that flexibility comes with a cost. Since there is no pre-defined architecture that developers can follow, it often leads to a messy, inconsistent, and over-complicated codebase."*
> — [Bulletproof React README](https://github.com/alan2207/bulletproof-react)

この柔軟性は、少人数・短期・小規模であれば開発速度を上げる長所になる。しかし本PJのように **25億円規模・複数チーム・複数サブドメイン・長期保守** が前提になると、構造を定めないことは直接的なリスクになる。

### アーキテクチャとは何か

アーキテクチャとは「コードをどのように分割・配置し、どこに何を書くか」というチーム全員が従うルールである。

Spring Boot における `Controller → Service → Repository` という3層構造がその典型例である。Reactにはこのデフォルトルールがないため、**チームとして明示的にアーキテクチャを選定・合意する必要がある。**

### アーキテクチャがない場合に何が起きるか

構造が定まらない状態で規模が拡大すると、以下の問題が発生する。

| 問題 | 内容 | 本PJへの影響 |
|---|---|---|
| **コード配置の不一致** | 開発者ごとに「どこに何を置くか」が異なる | 複数チームでコーディング規約が揺れ、コードレビューコストが増大する |
| **サブドメイン間の依存汚染** | 在庫管理のコードが発注管理に依存するなど、意図しない依存関係が生まれる | 一箇所の変更が別サブドメインに波及し、デグレを引き起こす |
| **オンボーディング遅延** | 構造が人によって異なり「どこを見ればよいかわからない」状態になる | 途中参画が多い本PJで、学習コストが開発速度を阻害する |
| **長期保守の劣化** | ルールがなければ時間とともに構造が崩れていく | 数年単位の保守が前提の本PJでコードベースの健全性が維持できない |

これらは「将来の話」ではない。アーキテクチャを定めなかった場合、以下のような出来事が**実際に発生する**。

---

**① コード配置の不一致 — 「同じ質問に3通りの答えが返ってくる」**

> 「在庫の絞り込みロジックはどこに書けばいいですか？」
> - Aさん：「`components/` に書きましたよ」
> - Bさん：「いや、`utils/` に置くべきでしょ」
> - Cさん：「自分は画面ファイルに直接書きました」

同じロジックが3箇所に散在する。バグを1箇所直しても、別の箇所に同じバグが残り続ける。

---

**② サブドメイン間の依存汚染 — 「触っていないはずの画面が壊れた」**

```typescript
// 発注チームのエンジニアが書いたコード（悪例）
import { formatProductCode } from '../../inventory/utils/format' // 在庫の内部関数を直接参照

// 在庫チームが formatProductCode をリネームした翌朝
// → 発注画面がビルドエラーで動かなくなった
// → 在庫チームは「内部ファイルのリネームをしただけ」という認識
// → 原因調査に半日、修正と動作確認にさらに半日
```

壊した側も壊れた側も、意図しないまま障害が発生する。チームをまたいだ変更のたびにこのリスクが生まれる。

---

**③ オンボーディング遅延 — 「全体を把握している人がいない」**

途中参画エンジニアが最初の1週間に直面する会話：

> 「ログイン処理はどこに実装されていますか？」
> 「`pages/Login.tsx` だと思いますが、一部は `hooks/useAuth.ts` にも。トークンの管理は `store/authSlice.ts` で… あと `api/auth.ts` もあって…」

設計者がいても「ちゃんと説明できない」構造になっている。初日から即戦力になれず、立ち上がりに1〜2週間かかる。

---

**④ 長期保守の劣化 — 「1年後の src/ フォルダ」**

プロジェクト開始1年後、新規参画エンジニアが `src/` を開くと：

```
src/
├── components/          # 400ファイル。どれが在庫でどれが発注か不明
├── utils/               # 「共通」のはずが機能固有の処理が混在
├── inventory.tsx        # 在庫管理のすべてが1ファイルに（1,200行）
├── order-new.tsx
├── order-new2.tsx       # なぜ "2" が存在するのか誰もわからない
└── order-new-final.tsx  # "final" が本番で動いている
```

誰も全体を把握できず、触ることへの恐怖からコードが肥大化し続ける。

---

これらを防ぐために、プロジェクト開始時点でアーキテクチャを選定・文書化し、チーム全体で合意する必要がある。

---

## 課題

フロントエンドのディレクトリ構成・ソフトウェアアーキテクチャが未決定であり、以下4点を同時に解決する構造が必要である。

1. **複数サブドメインの分離** — 在庫・発注・物流など独立したドメインを、互いに影響なく開発・保守できること

   > 在庫チームが `ProductCard` コンポーネントを修正したら、発注画面が壊れた—という事故が起きない構造。チームをまたいだ変更が互いに干渉しないことを、ルールではなく**仕組みで保証する**必要がある。

2. **大規模チームでのスケーラビリティ** — 複数チーム並行開発でコンフリクトが最小化され、画面数が増えても構造が破綻しないこと

   > 現在10画面でも機能する構造が、50画面・100画面になったとき「全員が同じフォルダを編集するコンフリクト地獄」にならないための設計。画面数の増加がそのまま複雑度の増加につながらない構造が必要である。

3. **立ち上げ速度と学習コスト** — 途中参画する多数のエンジニアが即座に構造を理解し、開発に参加できること

   > 25億規模のPJでは途中参画が常態化する。「在庫の検索条件はどこに書くの？」という質問に、チームの誰でも**同じ答えを即座に**返せる構造が必要である。答えが人によって異なる状態は、構造が定まっていない証拠である。

4. **バリデーションロジックの配置** — バックエンドのドメインルールと重複する入力検証を、どこに・どう管理するかが明確であること

   > 「商品コードは13桁」というルールはバックエンドに存在するが、入力時の即時フィードバックのためにフロントでも検証する。バックエンドがルールを変更したとき、フロントの検証コードが古いままになるリスクをどう管理するかが未解決である。

## 結論

**Feature-Based Architecture（Bulletproof React式）を採用する。**

[alan2207/bulletproof-react](https://github.com/alan2207/bulletproof-react)（GitHub 29k+ stars）が提唱する、**Feature（機能/サブドメイン）単位でコードを凝集させ、共通層と分離するハイブリッド構成**である。

### Bulletproof React式の設計原則

bulletproof-react は「何をもって React アプリを堅牢（bulletproof）と呼ぶか」を以下の原則で定義している。本PJのアーキテクチャはこの原則に基づいて選定している。

| 設計原則 | 意味 | 本PJへの対応 |
|---|---|---|
| **Easy to get started with** | 新規参画者がすぐに開発を始められる | `components/hooks/api` というReact開発者が日常的に使う分類のため構造説明が5分で終わる |
| **Simple to understand and maintain** | 読んで理解しやすく、長期的に保守しやすい | feature単位で凝集しているため「在庫管理に関するコードはすべて `features/inventory/` にある」と即座にわかる |
| **Clean boundaries between different parts** | コードの各部分の境界が明確 | `eslint-plugin-boundaries` でfeature間の不正importをESLintが静的に禁止する |
| **Everyone on the team is on the same page** | チーム全員が同じルールで開発できる | 本ADRとディレクトリ規約を文書化し、コードレビューチェックリストで運用する |
| **Scalable in terms of codebase and team size** | コードベースとチーム規模の両方でスケールする | サブドメインを追加する場合は `features/` 以下に新しいディレクトリを追加するだけ。既存コードへの影響はない |
| **Issues detectable as early as possible** | 問題をできるだけ早く検出できる | TypeScript strict モード + ESLint + Vitest により静的解析・単体テスト・境界チェックが常時実行される |

### ディレクトリ構成

```
src/
├── app/              # アプリ設定、プロバイダー、ルーティング
├── components/       # 共通UIコンポーネント
├── config/           # アプリ設定
├── features/         # ★ サブドメイン単位のモジュール
│   ├── inventory/    #   在庫管理
│   │   ├── api/      #     API呼び出し
│   │   ├── components/#    UI
│   │   ├── hooks/    #     ロジック
│   │   ├── schemas/  #     バリデーションスキーマ
│   │   ├── types/    #     型定義
│   │   └── index.ts  #     公開API
│   ├── orders/       #   発注
│   └── logistics/    #   物流
├── hooks/            # 共通フック
├── layouts/          # レイアウト
├── providers/        # グローバルプロバイダー
├── routes/           # ルーティング定義
├── types/            # 共通型定義
└── utils/            # 共通ユーティリティ
```

各 `features/` ディレクトリがバックエンドの境界づけられたコンテキスト（Bounded Context）と1対1で対応し、チーム割り当て・開発・保守の単位となる。

### コードの単方向フロー（Unidirectional Codebase）

Bulletproof React式が持つ重要な特性の一つが「コードの流れを一方向に制限する」ことである。Spring Boot の依存関係が `Controller → Service → Repository` の一方向であるように、フロントエンドのコードも一方向に制御する。

```
【コードの依存方向】

  shared（共通層）         どこからでも参照できる
  components / hooks / utils / types
         ↓  ↓  ↓
  features（機能層）       共通層のみ参照可。他featureへの直接参照は禁止
  features/inventory
  features/orders
  features/logistics
         ↓  ↓  ↓
  app（アプリ層）          featuresと共通層を組み合わせてアプリを構成する
  app/routes / app/provider
```

**なぜ一方向に制限するか：**

双方向の依存（「在庫がある発注の関数を呼ぶ」「発注が在庫の関数を呼ぶ」）が生まれると、一方を変更すると他方が壊れる連鎖が起きる。一方向に制限することで、各featureが完全に独立した「交換可能なモジュール」になる。

**ESLintによる自動強制：**

このルールは人の意識だけに頼らず、`eslint-plugin-boundaries` を使って静的解析で自動的に強制する。ルール違反はコンパイル時（コード保存時）に即座に検出され、コードレビューの前段で排除できる。

```
// ✅ 許可：共通層を参照する
import { Button } from '@/components/ui/button'
import { formatCurrency } from '@/utils/formatters'

// ❌ 禁止：別featureを直接参照する（ESLintがエラーを出す）
import { useOrderStatus } from '@/features/orders/hooks/use-order-status'
//  → ESLint: "Cross-feature import is not allowed"
```

各パターンの概要と不採用理由：

| パターン | 概要 | 本PJでの不採用理由 |
|---|---|---|
| **Traditional** | 技術種別（components/hooks/utils）でフラットに分割 | 50画面超で破綻する。サブドメイン境界が存在しない |
| **Atomic Design** | UI粒度（Atoms→Molecules→Organisms）で階層化 | UIの設計方法論であり、アプリ構造を規定しない。ドメイン分離不可 |
| **FSD** | 6層（app/pages/widgets/features/entities/shared）の厳格な依存方向制御 | entities層が活用されない（業務ロジックはバックエンド）。6層の学習・統制コストが立ち上げ速度を阻害 |
| **Clean Arch** | ドメイン層/アプリ層/インフラ層/プレゼン層に分離 | domain層・application層がほぼ空になる。Reactのコンポーネント指向との摩擦が大きい |
| **DDD Modular** | サブドメイン×4層（domain/application/infrastructure/presentation） | 構造が過剰。フロントのUI責務に対して4層は不要。採用実績がほぼ存在しない |
| **Micro Frontends** | サブドメインごとに独立したアプリとしてビルド・デプロイ（Module Federation / single-spa等） | 全サブドメインがReact+Vite+TSの統一スタックであり、技術スタック独立性のメリットが活きない。社内業務システムのため独立デプロイの優先度も低い。合成基盤（シェルアプリ・Module Federation設定・共有状態の協調）の構築・運用コストが大きく、共通デザインシステムの共有も困難になる |

## なぜ Bulletproof React式なのか

選定の根拠は4つに集約される。

### 1. Micro Frontendsほどの分離は不要

本PJの規模と複数サブドメイン構成を見ると、Micro Frontendsは有力候補に映る。IKEA・Spotify等の大規模実績もある。しかし本PJには以下の前提がある。

- **全サブドメインがReact + Vite + TypeScriptの統一スタック** — Micro Frontendsの最大のメリットである技術スタック独立性が活きない
- **社内業務システム** — 消費者向けプロダクトと異なり、サブドメインごとの独立デプロイ・独立リリースの優先度が低い。全サブドメインを一括デプロイしても運用上の問題が小さい
- **共通デザインシステムの重要性が高い** — 業務システムではUI/UXの統一性が生産性に直結する。Micro Frontendsではデザインシステムの共有に追加の仕組み（npm private package等）が必要になり、バージョン不整合のリスクが生まれる
- **合成基盤のコスト** — シェルアプリ・Module Federation設定・ルーティング協調・共有状態管理などのインフラ構築・維持にエンジニアリングコストがかかる

**feature分割で十分なドメイン分離が得られる本PJでは、Micro Frontendsはオーバーエンジニアリングにあたる。**

### 2. フロントの責務に対して「過不足がない」

本PJのフロントエンドは業務ロジックの正（Source of Truth）を持たない。ただし**入力バリデーション（桁数チェック・フォーマット変換・必須チェック等）はフロントの責務**として存在する。バックエンドのドメインルールの「検証コピー」を薄く持つ構造になる。

この前提で各パターンを見ると：

- **Clean Arch / DDD Modular** → domain層にバリデーションを置けるが、それ以外（集約・リポジトリ・ドメインサービス等）は**空になる**。バリデーションのためだけに4層構造を維持するのは過剰
- **FSD** → entities層にバリデーションを置けるが、entities本来の目的（ドメインモデルの表現）に対して**責務が薄すぎる**
- **Bulletproof React式** → feature内の `schemas/` にバリデーションスキーマ（Zod等）を配置する。**フロントの実際の責務（API呼び出し・バリデーション・状態管理・UI描画）に対して1対1で対応する構造**であり、空の層も過剰な層も生まれない

**フロントの責務が「表示 + バリデーション」であり「ドメインモデリング」ではないからこそ、feature内にschemasを置く軽量な構造が最適解になる。**

### 3. サブドメインとfeatureが自然に対応する

バックエンドのBounded Contextとフロントのfeatureディレクトリを1対1で対応させることで、チーム構造・コミュニケーション・コードの所有権が一貫する。

| バックエンド（ヘキサゴナル + DDD） | フロントエンド（Bulletproof React式） |
|---|---|
| Bounded Context（在庫・発注等） | `features/inventory/`, `features/orders/` |
| Application Service | feature内の `hooks/`（API呼び出し + 状態管理） |
| Port（入出力インターフェース） | feature内の `api/`（バックエンドAPI呼び出し） |
| Domain Model（バリデーション含む） | バックエンドが正。フロントは `schemas/` に検証コピーを持つ |

バックエンドとフロントで同じ「在庫」「発注」という言葉を使い、同じ境界でチームを分割できる。設計思想の一貫性を**無理な抽象化なしに**実現できる。

### 4. 大規模チームが「即日で」開発に参加できる

25億規模のPJでは途中参画が常態化する。アーキテクチャの学習コストは開発速度に直結する。

- **Bulletproof React式**: `components/hooks/api/types` というReact開発者が日常的に使う分類のため、**構造の説明が5分で終わる**
- **FSD**: 6層の責務と依存方向ルールの習得に**数日〜1週間**を要する
- **DDD Modular**: DDD概念（集約・値オブジェクト等）の理解が前提で、**フロントエンジニアには馴染みが薄い**

学習コストの低さは、チーム規模が大きいほど累積的な効果を生む。

---

## 結論（再掲）

**Feature-Based Architecture（Bulletproof React式）を採用する。**

本PJのフロントエンドは「業務ロジックの正はバックエンドに置きつつ、バリデーション・表示・UI/UXを責務とする」という明確な役割を持つ。この責務に対して、Bulletproof React式はfeature内にapi/components/hooks/schemas/typesを持つ過不足のない構造を提供する。サブドメイン分離・チーム並行開発・オンボーディング・バリデーション配置の4つの課題を、最小の構造的オーバーヘッドで解決できる唯一のパターンである。

### リスクと対策

| リスク | 対策 |
|---|---|
| feature間の依存が暗黙的に生まれる | `eslint-plugin-boundaries` でfeature間の直接importを静的に禁止 |
| featureの粒度判断がブレる | feature分割ガイドラインを文書化し、バックエンドのBounded Contextと整合させる |
| 構造の規律が長期的に緩む | コードレビューチェックリストに構造違反を含め、ADR運用を継続する |
